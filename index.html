Virtual DOM

是一个虚拟DOM树，用原生JS对象去描述一个DOM节点。借鉴了Snabbdom.js
Vue使用 Virtual DOM 来渲染组件，并且通过 diff算法 来比较 Virtual DOM 树的差异，从而减少操作 实际DOM

在Vue中，VNode是对 真实DOM 的抽象描述，有几个关键属性：标签名tag、数据data、子节点children、键值text、上下文环境context等
通过与真实 DOM 进行比较，可以找出需要更新的部分，从而减少了 DOM 操作次数，提高了性能

作用：

  • 维护视图和状态的关系，可以跟踪状态变化
  • 复杂视图情况下提升渲染性能
  • 可以跨平台渲染

除了数据结构等定义外，要映射到 真实的DOM节点 上实际上还需要几个步骤：

  1. create 创建Virtual DOM树
  2. diff
  3. patch 用于渲染对比VNode变成真实DOM

数据响应，数据劫持

实现由两部分组成，分别是观察者(watcher) 和 依赖收集器(Dep)

  • Vue 2.0响应 是基于ES5 的 Object.defineProperty 实现，在对象上定义一个新属性或修改对象的现有属性并返回此对象
    • 设置 data 后，遍历所有属性，利用 defineProperty 重写 属性的 getter 与 setter 方法，从而完成监听数据的改变
    • 缺点：无法监听数组变化；无法处理属性也为对象的情况；深度监听需要递归到底，一次性计算量大；无法监听新增/删除属性
  • Vue 3.0响应 是基于ES6 的 Proxy 实现，new Proxy(target, handler)
    • Proxy 对象用于创建 一个对象的代理，从而实现基本操作的拦截和自定义操作，如属性查找、赋值、枚举、函数调用等
      • 应用场景：校验器、私有属性…
    • 优点：不需要深度遍历监听，并且对象、数组变化都能监听得到(能观察的类型更丰富)
      • 但不兼容IE，也没有 polyfill, defineProperty 能支持到IE9
  • 两者都是对 对象上的某个属性 做处理
    • Object.definedProperty 是劫持 对象的属性，新增元素需要再次 definedProperty
    • Proxy 劫持的是 整个对象，不需要做特殊处理

重排重绘

repaint 重绘
当元素样式改变不影响布局时，浏览器将使用重绘对元素进行更新，此时只需要UI层面的重新绘制，损耗较少

  • 隐藏元素：v-show / visibility: hidden

reflow 回流/重排
当元素尺寸、结构或触发某些属性时，浏览器会重新渲染页面，此时浏览器需重新计算，计算后还需重新布局页面

  • 浏览器窗口大小改变
  • 元素尺寸、位置、内容改变 （v-if / 删除或添加可见的dom元素）
  • 元素字体大小改变
  • 激活css伪类 :hover
  • 某些属性或调用某些方法

    回流必定触发重绘，反之不一定
    重绘开销小，回流代价大

  • javascript方面
    • 避免频繁操作，可统一修改后再更新
    • 尽量使用class进行样式修改
    • 减少dom元素的增删修改

在浏览器中输入URL到展示的过程

可以大致描述为：
输入URL - 发起URL请求（DNS解析、建立TCP连接、发送请求、服务端处理请求并发送响应）- 浏览器接收响应 - 渲染页面

这一过程共涉及到3个进程，分别是浏览器进程、网络进程和渲染进程

  1. 处理输入，当在地址栏输入内容时，地址栏先判断是搜索查询还是请求URL，如果是搜索内容，会发送到搜索引擎合成新的带搜索关键字的URL。如果是域名会加上协议（如https）合成完整的url。
  2. 点击回车后，浏览器进程通过IPC把URL传给网络进程，网络进程获取到URL先去找缓存中是否有缓存资源，如果有就拦截请求直接返回资源。没有则请求DNS返回域名对应的IP地址和端口号。
  3. 建立TCP连接，连接建立后浏览器端构建请求行、请求头、请求正文等数据包，向服务器发送请求
  4. 服务端收到请求信息后处理构建响应数据发送回浏览器的网络进程（这一步后断开TCP连接）
  5. 网络进程接收到响应数据后进行解析，比如检查状态码、响应处理（重定向在这一步发生，获取URL重新来过）
  6. UI线程通知渲染进程开始渲染页面，渲染过程：
    1. 解析HTML转换为DOM树（就算编写错误也会被HTML规范优雅地处理掉）
    2. 加载子资源，包括图像、CSS和JS等外部资源
      • script标签会暂停HTMl文档的解析，必须加载、解析和执行JS代码后再恢复HTMl文档的解析
    3. 解析CSS创建CSSOM树（CSS对象模型）并确定每个dom节点的计算样式
    4. 主线程遍历DOM树和CSSOM树创建布局树
    5. 主线程遍历布局树来创建绘制记录和图层树（绘制记录有点像canvas规定了先画什么）
    6. 确定绘制顺序后，开始合成显示在屏幕上
  7. 提交导航，更新浏览器界面状态，包括地址栏、浏览器历史记录、网络安全状态等
  • beforeunload方法
    • 如果导航是从渲染器进程启动的，比如用户单击链接或JS运行window.location，则渲染器进程首先检查beforeunload处理程序。然后进行与浏览器进程启动导航相同的步骤。
  • TCP建立连接，客户端和服务端通过3次握手来确保双方能正确识别对方序列号，从而实现可靠的通信
    • 客户端发送SYN（同步序列号）包，告诉服务器希望建立连接并指定了一个初识序列号
    • 服务端确认客户端序列号，发送SYN-ACK包
    • 客户端接收上述包后，发送ack包确认服务器序列号也发送自己的序列号，表示连接已建立
  • 关闭TCP连接，通过4次挥手确保连接正常终止，避免数据丢失或不完整的情况
    • 客户端发送FIN包，表示完成数据发送并不再需要连接（这个包中也包含了一个序列号）
    • 服务端回复ACK包确认收到客户端的FIN包
    • 服务端发送FIN包表示准备好关闭连接了（同样也包含了一个序列号）
    • 客户端回复ACK包确认了服务器序列号，表示接受到关闭连接的请求
  • 如果导航为重定向跨站点，则可能不会使用此备用进程，在这种情况下可能需要不同的进程
  • DNS是域名系统，互联网中将域名转换为IP地址的分布式数据库系统

安全性

  • XSS攻击，跨站脚本攻击
    • 目标是向网页中注入恶意脚本代码，从而在用户浏览页面时执行这些恶意代码
    • 通常分为三种：
      1. 存储型XSS攻击，恶意脚本代码存储在网站数据库中，当访问页面时，会从服务端返回并执行
      2. 反射型XSS攻击，将恶意链接发送给用户，用户点击后返回恶意代码并执行
      3. DOM型XSS攻击，构造特定的URL或其他用户可控制的输入，当访问页面时，代码执行
    • 防御措施：
      • cookie设置httpOnly
      • 对用户输入进行严格过滤，转译页面上的输入内容和输出内容
      • 限制JavaScript执行的权限
  • CSRF(Cross-Site Request Forgery)跨站请求伪造
    • 在登录过的网站上进行非预期无感知的操作，比如向目标网站发送恶意请求，从而可以发起类似资金转账、更改账号信息等未授权的操作
    • 防止攻击的常见方法：
      1. 使用CSRF Token，随机生成Token，并在服务端检验有效性，确保请求合法
      2. 同源检测
      3. 使用 SameSite Cookie属性，限制Cookie发送，防止跨站点请求
      4. 设置合适的HTTP头部（X-Ruquested-With、Content-Type…）确保请求安全性
      5. 使用双重身份验证，增加攻击者难度
      6. 设置白名单，不被第三方网站请求；不被第三方网站访问到用户的cookie

存储机制

一般主流方案：Cookie、Web Storage、IndexedDB、Cache Storage

Cookie，通过浏览器在客户端存储少量数据

  • 存储量较小，通常为4kb大小；
  • 可以被浏览器设置，每次请求都会自动携带，包含在请求头上，不安全
  • 也可以被服务器设置
  • 使用场景比较有限，通常用于存储用户信息，登录状态等

Web Storage，在客户端保存键值对数据，限制为4～5MB

  • LocalStorage，在浏览器关闭后仍然保存，直到手动删除。使用方便
  • SessionStorage，在当前会话结束时即被清除

IndexedDB，一种高级的客户端存储技术，允许在浏览器中存储大量结构化数据并支持查询

  • 存储容量大，支持事务，可高效地处理大量数据
  • API相对复杂，不同浏览器兼容性不同

Cache Storage，浏览器提供的一种缓存机制

  • 用于存储请求过的资源，如CSS、JS、图片等
  • 提高页面加载速度，减少网络请求，提升用户体验

  • 使用事件委托来减少事件处理程序的数量

跨域

是浏览器基于同源策略的一种安全手段，是对安全性做的限制

同源策略： 是一种约定，用于隔离潜在恶意文件的安全机制，通常不允许不同源之间的读操作

同源三要素：协议protocol / 主机host / 端口port 相同
非同源的请求就会产生跨域

常见于以下场景：

  • ajax跨域请求
  • 跨文档消息通信，一个页面通过postMessage发送消息到另一个页面时，如果两个页面的域名不同，则会触发跨域问题
  • 嵌套的跨域资源，如iframe、img、script
    解决方案：
  1. JSONP（JSON with Padding）动态创建script标签来加载脚本，实现跨域数据传递
    • 可兼容老式浏览器
    • 需要服务端支持且只支持GET请求
  2. CORS（Cross-Origin Resource Sharing）服务端设置响应头Access-Control-Allow-Origin: * | <origin>，允许指定的域名访问资源
    • 支持所有请求方法，但兼容性比JSONP差
    • 实现关键在服务端
  3. 代理服务器，在服务端设置代理服务器，由代理服务器转发请求，实现跨域访问。（node中间件）
  4. 反向代理，将需要访问的资源部署在同一域名下，通过反向代理将请求转发到真实的资源地址。（Nginx）
    • node中间件和Nginx反向代理都利用了服务器对服务器无同源策略
  5. iframe嵌套通信，document.domain + iframe / window.name + iframe / location.hash + iframe
  6. 基于OPTIONS方法进行过滤，发送一个预检请求到服务器，是否允许其实际请求
  7. postMessage，是HTML5 XMLHttpRequest Level 2中的API，可以实现跨文档通信
  8. webSocket，是HTML5的一个持久化协议，不受同源策略限制，可以建立跨域连接，可与postMessage配合使用。是一个长连接long poll

代理服务器 与 反向代理服务器 区别：

  1. 代理服务器位于客户端和原始服务端间，而反向代理服务器位于客户端和多个原始服务器间
  2. 代理服务器主要用于隐藏客户端的真实IP地址、提高访问速度、过滤内容等
  3. 反向代理服务器主要用于负载均衡、安全性、隐藏服务器架构等方面
  4. 代理服务器的配置由客户端控制，需手动配置代理服务器的地址和端口
  5. 反向代理服务器的配置由服务端控制，客户端无需感知反向代理服务器的存在

实际应用中，反向代理服务器更为常见。
特别是在构建高可用性、高性能的Web应用时。
通过反向代理，可以将请求分发给不同的服务器，实现负载均衡和故障转移；
同时，反向代理也可以用于缓存静态内容，提高访问速度；
此外，反向代理还可以作为防火墙，过滤恶意请求，增强系统的安全性。

Node的Event Loop事件循环

Timers -> Pending callbacks -> Idle/Prepare -> IO Poll -> Check -> Closing callbacks

  • Timers 用于存储定时器的回调函数(setInterval、setTimeout)

  • Pending callbacks 执行与操作系统相关的回调函数，比如启动服务器端应用监听端口操作

  • Idle/Prepare 系统内部使用，可以不关注

  • IO Poll 存储I/O操作的回调函数队列，例如文件读写操作

    • 如果事件队列中无回调函数，但 Check阶段 或T imers队列 中有需执行的回调函数
      • 那么事件循环会移至 Check阶段 或 Closing callbacks 并从 Timers 进入下一次循环
      • 否则会在IO Poll阶段停留一段时间以等待新的回调函数进入

  • Check 存储setImmediate API 的回调函数

  • Closing callbacks 执行与关闭事件相关回调函数，比如关闭数据库连接

  • Node适合I/O密集型任务，不适合CPU密集型任务，因为主线程一旦阻塞，就会卡程序

  • 宏任务：setInterval、setTimeout、I/O、setImmediate

  • 微任务：Promist.than/catch/finally、process.nextTick

  • 宏任务微任务各自的回调函数放在各自的任务队列中

  • 微任务优先级高于宏任务

  • 微任务中的nextTick高于其他微任务，先进先出

HTTP请求方法

get / post / options / put / patch / delete / head / trace

  • get，用于向服务器请求获取数据，如获取网页、图片等
    • 请求参数附在URL上，以查询字符串形式发送
    • 请求长度受限，不修改资源，无副作用
    • 可被缓存，也可以被书签、历史记录保存
    • 数据在URL中可见，不适合发送敏感信息
  • post，用于向服务器提交数据，如表单数据、上传文件等操作；更多编码格式
    • 请求参数附在请求体中，不会在浏览器历史记录中显示
    • 不可被浏览器缓存，不会被书签、历史记录保存
    • 数据不暴露在URL中，适合发送大量数据和敏感信息
  • head，类似于get，但服务器只返回请求头部信息，通常用于检查资源的元数据，例如文件是否存在、最后修改时间
  • options，用于请求服务器告知客户端对指定资源支持的各种请求方法、HTTP头部信息等
  • put，用于向服务器发送数据，从而更新指定资源内容。如果资源不存在，则创建一个新资源
  • patch，类似于put，但仅传输需要更新的数据部分，非整个资源
  • delete，用于请求服务器删除指定的资源
  • trace，用于在请求链路中追踪请求的传输路径，常用于诊断和调试

进程、线程

  • 进程是CPU资源分配等最小单位，是应用程序运行时的实例对象；进程间相互独立不受干扰；可以通过IPC来实现通信
  • 线程是CPU调度等最小单位，被包裹在进程中，应用程序具体要做的每件事情生成了一个个线程，类似于一个待办列表
  • 线程上下文切换比进程上下文切换要快得多
  • CPU可以有很多进程，但资源是固定的。进程越多，CPU分出去的资源越多就越卡。

JS为什么是单线程

这是由起初设计的用途和运行环境所决定

  • 起初被设计用于web页面互动及操作DOM，如果是多线程，会带来复杂的并发问题，如死锁（两个线程相互依赖）、同步问题、竞态条件（多个线程访问共享资源引起的不确定性和不可预测的行为）等
  • 多个js脚本同时修改页面可能会引起页面冲突，单线程保证了脚本顺序执行
  • 在每个页面中多线程可能会导致页面响应变慢或崩溃
  • 可以通过事件循环机制，处理异步任务，如网络请求、定时器等

web worker，是HTMl5提供的JS线程

  1. 独立线程，与主线程相互独立，不阻塞主线程。可并行执行多个任务
  2. 无法访问和操作DOM元素
  3. 支持多线程通信
  4. 受同源限制，无法加载本地资源
    使用场景：需要大量计算、网络请求等耗时的任务

层叠上下文

HTML元素沿着面向用户的一条虚构的z轴顺序排开，是对HTML元素的一个三维构想
运用z-index，将元素提升为一个比较特殊的图层，在三维空间(z轴)中高出普通元素一等

  • z-index属性设置元素的堆叠顺序
  • 拥有更高堆叠顺序的元素总是会处于顺序较低的元素的前面
  • z-index仅能在定位元素上奏效（position属性值不为static以外的元素，包括relative，absolute，fixed样式）

满足以下任意一个条件的元素就会形成层叠上下文:

  • html根元素
  • position为absolute / relative且z-index不为auto
  • position为fixed / sticky
  • 网格布局grid或弹性布局flex容器里的子元素且z-index不为auto
  • opacity属性值小于1的元素
  • …
  • 层叠上下文可以包含在其他层叠上下文中，并且一起创建一个层叠上下文的层级
  • 每个层叠上下文都完全独立于它的兄弟元素（当处理层叠时只考虑子元素）
  • 每个层叠上下文都是自包含的（当一个元素的内容发生层叠后，该元素将被作为整体在父级层叠上下文中按顺序进行层叠）

positon 该属性用于指定一个元素在文档中的定位方式

  • static：正常布局，即元素在文档常规流中当前的布局位置。top、left、right、bottom、z-index无效，按正常文档流进行排列
  • relative：元素不脱离文档流，在原始位置上不改变页面布局的前提下调整元素位置，不影响其他元素偏移。未添加定位时的位置会多出一块空白
  • sticky：元素根据正常文档流进行定位，然后相对他的最近的滚动祖先和块级祖先进行偏移，偏移值不影响任何其他元素位置
  • absolute：元素会被移出正常文档流，不预留空间，可设置外边距，不与其他边距合并
  • fixed：元素会被移出正常文档流，不预留空间，通过指定元素相对于屏幕视口的位置来指定元素位置，位置随屏幕滚动时不会改动，所固定参照对象是可视窗口
  • absolute的containing block计算方式跟正常流有什么不同？
    • 正常流中，元素的containing block是其最近的块级祖先元素
    • 但绝对定位absolute可能会是以下几种：
      • position: relative / absolute的祖先元素
      • 离它最近的包含块containing block，即最近的块级祖先元素
      • initial containing block，即viewport（如果都没找到符合的祖先元素）

viewport是指浏览器中用来显示网页内容的区域，即用户能看到的可视区域。随浏览器窗口大小或缩放操作而改变

为什么会产生浮动

JavaScript 内存与性能

V8，Google开发的JS引擎
V8将内存分为：

  • 新生代，存放生命周期较短的对象，采用Scavenge算法垃圾回收
  • 老生代，存放生命周期较短的对象，采用标记清除和标记整理等算法垃圾回收

垃圾回收机制，负责管理和释放不再被引用的内存

  1. Scavenge算法
    • 将内存空间一分为二，分为From空间 跟 To空间；From空间存放活对象
    • 当From空间被占满时，启动垃圾回收，将存活的对象复制到To空间，进行内存整理，From空间和To空间互换
  2. 标记清除，主流垃圾收集算法
    • 给存储在内存中的所有变量加上标记
    • 去除环境中及被环境中的变量所引用的变量的标记
    • 剩下标记的变量就被视为准备删除的变量

会产生内存碎片，导致内存空间不连续性

  1. 标记整理，标记清除改进版
    • 清理不再被引用对象之后，会对存活的对象进行整理，移动内存空间一端，减少内存碎片产生

  2. 引用计数，不太常见

  • 如果一个值的引用次数为0时，就会被释放内存
  • 函数被反复多次调用，就会导致大量内存得不到回收
  • 循环引用会导致问题，可能会导致内存永远不会被回收 ？

管理内存

  • 为执行中的代码只保存必要的数据
  • 尽可能少未声明就初始化变量，会变为全局变量
  • 解除引用：不仅可以消除循环引用，对垃圾回收也有帮助
    • 设置为null来释放其引用，让值脱离执行环境
    • 局部变量会在离开执行环境时自动被解除引用

性能问题

  • let、const都以块为作用域，相比于使用var可能会更早地让垃圾回收程序介入，尽早释放内存

  • 隐藏类和内嵌缓存来提高性能

  • 尽量做到及时回收不用的内存

    内存泄漏
    一些常见的内存泄漏情况：

  1. 意外的全局变量在页面声明周期内不会被释放
  2. 定时器未清理会导致函数或对象一直存在在内存中无法被回收
  3. 事件监听器未被正确移除，仍然存在在内存中
  4. 如果闭包中引用了大量变量而又未被及时释放
  5. dom引用：如果在不需要时未释放这些引用，导致dom元素无法被回收
  6. 循环引用无法被垃圾回收
  7. 在内存中缓存大量数据未及时清理

怎么避免呢？

针对上述的情况，可以多注意管理事件监听器、定时器等，及时释放不需要的资源

  • 合理使用变量和对象
  • 避免循环引用
  • 优化算法和数据结构
  • 及时释放资源，如打开文件、网络请求等，务必在使用完毕后及时关闭或释放资源
    还可以通过工具来监控和诊断问题，比如Google浏览器有一些工具

可以学习对Performance、Memory的运用来检测和分析内存泄漏
可以使用timeline进行内存标记，可视化查看内存的变量情况

JS性能
当谈到这个话题时，可以考虑几个方面：

  1. 运行时间，通过减少不必要的循环、尽可能少操纵DOM等方式优化JS代码的执行速度，使得页面加载速度更快，提升用户体验

  2. 内存占用也是一个关键指标，过度使用内存可能会导致页面加载慢、卡顿甚至崩溃；优化内存使用可以通过避免内存泄漏

  3. 渲染性能，频繁地操作DOM会导致重绘或重排，从而降低性能

  4. 网络性能

  5. 跨浏览器性能，在不同浏览器和设备上表现可能有所不同

    可以考虑采取以下措施提高性能：

  • 使用局部变量来提高访问速度
  • 避免使用eval函数，因为它会影响性能
  • 使用事件委托来减少事件处理程序的数量
  • 减少DOM操作次数，避免频繁的重绘和重排；使用批量更新
  • 使用合适的数据结构和算法
  • 使用异步加载来延迟不必要的资源加载
  • 缓存重复的计算结果
  • 使用Web Workers进行多线程处理，提高性能

      闭包

一个引用了另一个函数作用域中变量的函数
常见方式：在一个函数内部创建另一个函数
是一种静态作用域

  • 闭包作用域链包含自己的、包含函数的和全局的
    • 在作用域链中查找变量 (作用域链本质上是一个指向变量对象的指针列表)
  • 一般来讲，函数执行完毕后，局部活动对象就会被销毁，内存中仅保存全局作用域
  • 而闭包则会使内部定义的函数将包含函数的活动对象添加到它的作用域链中
    • 当被引用的函数返回后，其执行环境的作用域链会被销毁，但活动对象仍会留在内存中
  • 闭包只能取得包含函数中任何变量的最后一个值，因为它们引用的是一个活动对象里的变量
    • 闭包保存的是整个变量对象，而不是某个特殊变量
  • 匿名函数的执行环境具有全局性，this通常指向window
    • 要想访问作用域中的arguments对象，必须将该对象的引用保存到另一个闭包能访问的变量中
  • 闭包还可以用于在对象中创建私有变量、私有函数
    • 有权访问私有变量的共有方法叫 特权方法
      • 单例模式、构造函数模式、原型模式
    • 在原型prototype上定义公有方法 例如get() set()
      • 但在一个实例上调用时会影响所有实例，可能会导致返回相同的值
  • 闭包会引用包含函数的整个活动对象，有可能会造成引用无法解除，占用内存
    • 创建闭包必须要维护额外的作用域，过度使用可能会占用大量内存
  • 过度使用闭包可能导致内存占用过多

解决：

  • 变量可以通过函数参数的形式传入，避免使用默认的[[scope]]向上查找
  • 使用setTimeout包裹，通过第三个参数传入
  • 使用块级作用域，让变量成为自己上下文的属性，避免共享
  • 避免循环引用，确保不再需要时将外部函数的变量引用置为null

      • 实例和原型中的属性都包括（能通过对象访问的、可枚举的属性）

    • Object.getOwnPropertyNames()无论是否枚举都返回
  • 使用 对象字面量形式 创建时，相当于重写了prototype对象，constructor属性就会变成新的，指向了Object构造函数

原型陷阱: 当重写了整个原型对象时，就切断了跟先前已经创建的实例对象之间的联系，已经创建的实例引用的依然是最初的原型

  • 当原型对象中包含引用类型值属性时，当通过某个实例改变值时会导致所有实例指向的该属性值都会改变
  • 推荐使用构造函数 + 原型模式组合使用
  • 寄生构造函数模式，稳妥构造函数模式（安全性）

是一个用来实现继承和共享属性的对象链

尾调用优化

  • 尾调用就是一个出现在另一个函数“结尾”处的函数调用（某个函数最后一步是调用另一个函数）
  • 函数调用会形成一个调用记录（调用帧），所有的形成一个调用栈
    • 当出现尾调用时，如函数A内最后调用了B，此时A就结束了，那完全可以删除A的调用帧，只保留B的调用帧即可
    • 每次执行时调用帧只有一项，将大大节省内存
    • 但只有不再用到外层函数的内部变量时，才能进行尾调用优化（闭包就不行）
  • 在递归问题上可以更好地运用起来，即尾调用自身，只保留一个调用记录，复杂度为O(1)
  • ES6中尾递归不会发生栈溢出，相对节省内存

模块

简单来说就是将代码分成独立的块，再把这些块连接起来
思路：逻辑分块、各自封装、相互独立，每个块自行决定对外暴露什么，同时也自行决定引入哪些外部代码。

模块必须在依赖加载完成后才能被加载
ES6之前可以使用 函数作用域 和 立即调用函数表达式 将模块定义封装在匿名闭包中
模块永远是单例，无论请求多少次，只会被加载一次

  • CommonJS
    • 在服务器端实现模块化代码组织
    • Node.js 使用了轻微修改版本的 CommonJS，其中它不需要考虑网络延迟问题
    • 使用require(..)指定依赖，用module.exports定义公共API
  • AMD
    • 以浏览器为目标执行环境，需要考虑网络延迟问题
    • 实现核心是用函数包装模块定义，可以防止声明全局变量
    • define(..)定义公共API，异步加载
  • UMD，通用模块定义
    • 统一了CommonJS和AMD生态系统，在启动时检测要使用哪种模块系统，然后进行适当配置，将逻辑包装在一个立即调用的函数表达式里
  • ES6模块
    • <script type=”module”></script>
    • 是异步加载和执行的
    • 顶级this 的值是 undefined（windows下）
    • 解析到上述标签后回立即下载模块文件，但执行回延迟到文档解析完成；有多个标签时也是按顺序执行
    • 嵌入模块的代码不能使用 import 加载到其他模块。只有通过外部文件加载的模块才可以使用 import 加载
    • 使用 export 导出，且必须在模块顶级，不能嵌套在语句中。支持 命名导出 和 默认导出
      • 默认导出使用 default关键字 作为值来导出，default命令只能使用一次，不可重复默认导出。不用加{ }
      • 输出的值是动态绑定的，绑定其所在模块
    • 使用 import 引入其他模块导出的值，也必须出现在模块顶级，不能嵌套
      • 会被提升到模块顶部
      • 导入对模块而言是只读的，相当于是const声明的变量，不能直接修改导出值

总结：

  • CommonJs，由Node.js社区采用
    • require导入模块，module.exports 和 exports 导出模块
    • 在运行时同步加载模块，适用于服务器端（运行时加载）
    • 模块输出的是值的拷贝，导出值变化不会影响导入值
    • 模块就是对象，输入时必须查找对象属性
    • 加载整个模块里的所有方法，然后在使用时再运用对应的方法
  • AMD，RequireJS 库提出
    • require异步加载模块，define定义模块
    • 适用于浏览器端，允许异步加载依赖模块
  • ES6 Modules，ES6中的模块化规范
    • import导入模块，export导出模块
    • 按需加载 异步加载 编译时加载
    • 语法更清晰
    • 输出的是值的引用，import指向内存地址，导入值会随导出值而变化

        • 根据具体情况，有时 Promise 可能会更容易理解和维护，因为它提供了一种线性的、顺序执行的方式来处理异步任务

前端存储总结

一般的主流方案有三个：

  • Cookie
  • Web Storage

Cookie

1994年被发明，并不是为了在 浏览中存放数据，是为了解决 HTTP协议无状态特性 的问题

HTTP协议无状态特性：用户两次HTTP请求，服务器并不能知道请求来自同一个用户

现在仍在部分领域有价值，但不再适合作为前端本地存储的方案去使用

  • Cookie安全问题

    • 每次请求都会发送Cookie，因为如果不是HTTPS协议并加密，其保存的信息容易被窃取，导致安全风险
    • 一旦信息被窃取，容易被利用，信息泄露
    • 可以用session配合缓解问题，但session同时会占用额外的服务器资源
    • 每次请求自动发送 这个特性会导致CSRF攻击的安全风险
    • 可能还会增加带宽占用，但依据现在的网络环境可以忽略不计

  • Cookie只允许存储4kb大小的数据

  • 操作较复杂，但可以使用类库来解决

常用的有在广告中，跨站标记用户、跟踪用户行为
例如你打开了两个页面，但能看到同样的广告

Cookie 执行同源策略，这里就衍生了第三方Cookie
例如访问 a 时，设置了 b 的 Cookie ，那么对 a 来说就是 第三方Cookie
第三方Cookie有一个重要特性：Cookie可以被服务端通过请求头来要求浏览器设置

画图描述
当用户处于 abc.com 时，浏览器会向 taobao.com/some-ads 发起一个 HTTP 请求
当淘宝服务器返回广告内容时，会顺带一个 Set-Cookie 的 HTTP 请求头，告诉浏览器设置一个源为 taobao.com 的 Cookie，里面存上当前用户的 ID 等信息
这个 Cookie 相对于 abc.com 而言就是第三方 Cookie，因为它属于 taobao.com
而当用户访问 xyz.com 时，由于 xyz.com 上也嵌入了淘宝的广告，因此用户的浏览器也会向 taobao.com/some-ads 发起请求
有意思的来了，发请求时，浏览器发现本地已有 taobao.com 的 Cookie（此前访问 abc.com 时设置的），因此，浏览器会将这个 Cookie 发送过去
淘宝服务器根据发过来的 Cookie，发现当前访问 xyz.com 的用户和之前访问 abc.com 的用户是同一个，因此会返回相同的广告

综上，Cookie 两个特点：可以被浏览器设置、浏览器每次请求会自动带上

  • Cookie设置

    • SameSite：Chrome 80版本新特性，有三个值：Lax（允许部分）/None（都发送）/Strict（不发送）
      默认设置为Lax，即允许同站或子站访问Cookie，目前支持

      get表单
      老版本则是None，允许所有跨站Cookie
      设置Lax可能会导致第三方Cookie失效
      解决：在返回请求的header里设置 Set-Cookie: userId=123; SameSite=None

    • Secure，HttpOnly：常用属性
      Secure只允许 Cookie 在 HTTPS 请求中被使用
      HttpOnly 禁止使用 JS 访问 cookie，这样的好处就是避免了XSS攻击
      配置如下：Set-Cookie: userId=123; SameSite=None; Secure; HttpOnly

  • XSS攻击
    例如一些网站不会对内容的HTML标签进行过滤，那么就可以对其发起攻击或者在console可以用document.cookie获取到cookie信息
    当设置了 HttpOnly 后，以上的做法就失效了

总结：使用场景比较有限，由于安全性和储存空间的问题，一般不推荐使用 Cookie

Web Storage

2014年发布的HTML5标准中，新增了Web Storage，包括LocalStorage、SessionStorage

LocalStorage特点：

  • 使用 Key-Value 形式储存，key、value以字符串存储
    只能以字符串，这个问题可以看这个 https://www.zhihu.com/question/366665675/answer/1127726009
  • 大小有 10MB
  • 使用方便

跟Cookie对比起来：

  • 存储空间更大，使用更方便
  • Cookie可以被服务器设置，但LocalStorage只能被前端手动设置
  • Cookie会由浏览器自动发给服务器，
    LocalStorage需要手动获取放到请求里发送给服务器，因此可以避免 CSRF 攻击

CSRF攻击，跨站伪造攻击

  • 特点：诱导你去访问某个需要你的权限的接口，HTTPS接口并不能阻止这种访问
    某些恶意网站会诱导你点击表单按钮，请求一个你之前登录过的网站的接口，此时浏览器会将你的Cookie一并发送

用LocalStorage替换 比较常见的做法是：
token 加上HTTPS请求，提高登录安全性
即 用户登录后，将服务器返回的 token存到 LocalStorage 里，
之后每次请求前都取出 token放到请求数据里，
服务器就知道是同一个用户发起的请求
由于 HTTPS 的存在，也不用担心 token 会被泄露给第三方

  • 缺点：

    • 无法像Cookie设置过期时间

    • 只能存入字符串，一般使用 JSON.stringify() 配合 JSON.parse() 解决

      但 JSON.stringify() 本身存在一些问题，即无法正确转换JS部分属性，无法转换循环引用的对象
      undefined, Function, RegExp(转换后变成了空对象）, Date(转换后变成了字符串)

      Symbol 无法被转换，
      但由于 Symbol 本身的定义（全局唯一性）就决定了它不应该被转换，否则即使转换回来，也不会是原来的Symbol

      Function 比较特殊，可以先调用 .toString() 转换为字符串储存，需要时再 eval 转回来

      总结：大部分场景下 LocalStorage 可以替代 Cookie ，但最多支持4～5MB，只支持字符串

      不适用 大量数据 和 复杂数据 的存储

JS

js运行机制

看到这个问题，首先想到了什么？单线程？任务队列？

进程

CPU资源分配的最小单位，可独立运行且拥有自己的资源空间的一个任务程序，包括运行中的程序和程序所使用到的内存和系统资源

  • 进程之间相互独立

    • 但同一进程下各个线程共享程序的内存空间(包括代码段、数据集、堆等)及进程级的资源(如打开文件和信号)

  • 多进程的好处是可以同时干不同的事情，且不受干扰

  • 如果进程间需要通信，可以通过IPC(进程间通信)来实现

线程

CPU调度的最小单位，即程序执行的最小单位。

  • 建立在进程基础上的一次程序运行单位，即可以理解为程序中的一个执行流，一个进程可以有多个线程

单线程，就是一个进程中只有一个执行流。反之有多个就叫多线程，允许单个程序创建多个并行执行的线程来完成各自的任务

CPU可以有很多进程，但资源是固定的，进程越多，CPU分出去的资源越多，越卡。
在运行一个进程时，其他进程处于非运行状态，使用时间片轮转调度算法

线程上下文切换比进程上下文切换要快得多

JS为何是单线程

主要与用途有关: js用途主要是与用户互动及操作DOM，如果不是单线程会带来复杂的同步问题

web worker

  • JS的异步特性本质上还是绑定在一个单事件循环线程上。
  • HTML5 提出Web Worker标准，允许js脚本创建多个线程，但子线程完全受主线程控制，不得操作DOM
  • 可以在独立的线程运行一个JavaScript文件(即程序)，使用异步事件在线程之间传递消息。

浏览器多线程

  • Browser进程
    • 浏览器的主进程(负责协调、主控)，该进程只有一个
    • 负责浏览器界面显示，与用户交互。如前进，后退等
    • 负责各个页面的管理，创建和销毁其他进程
    • 将渲染(Renderer)进程得到的内存中的Bitmap(位图)，绘制到用户界面上
    • 网络资源的管理，下载等
  • 第三方插件进程
    • 每种类型的插件对应一个进程，当使用该插件时才创建
  • GPU进程
    • 该进程也只有一个，用于3D绘制等等
  • 渲染进程（重点）
    • 即通常所说的浏览器内核(Renderer进程，内部是多线程)
    • 每个Tab页面都有一个渲染进程，互不影响
    • 主要作用为页面渲染，脚本执行，事件处理等
  • 浏览器如果为单进程，那么任何一个tab崩溃或者插件奔溃就会影响到整个浏览器，这样的体验是很大的
  • 但浏览器进程很多，每个进程又有很多线程，都会占用内存。这意味着内存等资源消耗会很大，有点类似空间换时间的意思

Renderer进程

一些主要线程

  • GUI渲染线程

    • 解析HTML、CSS、构建DOM树和RenderObject树，布局和绘制等
      • 解析html代码（本质是字符串）转化为浏览器的节点，生成DOM Tree
      • 解析css，生成CSSOM（css规则树）
      • DOM Tree和 CSSOM 结合，生成Rendering Tree
    • 重绘 Repaint
    • 回流 Reflow
    • Repaint和Reflow时GUI线程执行
    • Reflow 比 Repaint 成本要高，尽量避免这两者

  • JS引擎线程

    • 是JS内核，负责处理JS脚本程序，例如V8
    • 解析JS脚本，运行代码
    • 一直等任务队列中的任务然后开始处理
      • 浏览器同时只能有一个JS引擎运行JS程序，所以JS单线程运行
      • 一个Tab页无论何时只有一个JS线程

  • 事件触发线程

    • 属于浏览器而不是JS引擎，用来控制时间循环，管理一个事件队列 task queue
    • 只关心异步回调

  • 定时触发器线程

    • setTimeout、setInterval所在线程
    • 由于JS引擎单线程，如果处于阻塞线程状态就会影响计时的准确，所以并不由JS引擎计数
    • 通过单独线程来计时并触发定时，计时完毕后，添加到事件触发线程的task queue中，等待JS引擎空闲后执行
    • W3C在HTML标准中规定，要求setTimeout中低于4ms的时间间隔算为4ms
    • 只管理定时器不关心结果，结果在事件触发线程中

  • 异步http请求线程

    • XMLHttpRequest 在连接后通过浏览器新开一个线程请求
    • 简单地说，就是当执行到一个异步请求，就将异步请求事件添加到异步请求线程，等待http状态变化，再将回调函数添加到task queue中，等待JS引擎执行

GUI渲染线程和JS引擎线程是互斥的
JS引擎会阻塞GUI渲染引擎，此时GUI线程会被挂起，可以认为被冻结了

  • 也就是JS执行时间过长就会导致页面渲染加载阻塞，加载慢
  • 例如浏览器渲染的时候遇到script标签，就会停止GUI的渲染，然后js引擎线程开始执行js代码，等js执行完毕，
    js引擎线程停止工作，GUI继续渲染。所以如果js执行时间太长就会造成页面卡顿的情况
  • GUI更新会被保存在一个队列中等到JS引擎空闲时立即被执行

那么，上面都提到了一个共同点
来看看这个到底是什么？

事件循环

JS 有 同步任务 和 异步任务
同步任务在主线程上执行，形成一个执行栈

上面说到，事件触发线程管理一个事件队列 task queue
只要异步任务有运行结果，就在任务队列中放一个事件回调

一旦执行栈中的所有同步任务执行完毕
即JS引擎线程空闲下来了，系统就会读取任务队列
将可运行的异步任务添加到执行栈中

宏任务 微任务

宏任务：macrotask 也被称为 task

浏览器为了使 宏任务 和 DOM任务 有序地进行，通常是下面的顺序
宏任务 -> GUI渲染 -> 宏任务

常见的宏任务

  • 主代码块
  • setTimeout
  • setInterval
  • setImmediate() -> Node
  • requestAnimationFrame() -> 浏览器

微任务：microtask
ES6新引入了Promise标准，同时浏览器实现上多了一个微任务概念
在ECMAScript中，microtask 也被称为 jobs
可以看作是宏任务执行后立即执行的任务

常见微任务

  • process.nextTick() - Node
  • Promise.then()
  • catch
  • finally
  • Object.observe
  • MutationObserver

当一个宏任务执行完，会在渲染前，将执行期间所产生的所有微任务都执行完
是这样的执行顺序
宏任务 -> 微任务 -> GUI渲染 -> 宏任务

new Promise(() => {}).then()

new Promise(() => {}) 这部分是一个构造函数，是一个同步任务
.then() 是 异步微任务

async/await 本质上还是基于promise的封装
