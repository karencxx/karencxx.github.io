  1. CSS选择器的优先级是如何计算的？
    css选择器优先级为 内联样式 > ID选择器 > 类/伪类/属性选择器 > 元素和伪元素选择器 > 通配符选择器/继承 > 全局/默认样式

    每个选择器类型有不同的权重，优先级按从左到右逐级计算，优先级高的会覆盖优先级低低

JavaScript

  1. let、const和var的区别？
    在作用域、提升行为和可变性有所不同
    let和const是块级作用域，会产生变量提升，但由于暂时性死区，在声明前不能被访问到，在块级作用域内只能声明一次，let可以重复赋值，但const在声明时必须赋值且不可再次赋值，如果是对象，那么只是引用地址不变，可以添加新值
    var的作用域是函数作用域和全局作用域，存在声明提升，可重复声明赋值

  2. 什么是闭包，闭包的应用场景有哪些？
    闭包简单地来说是在函数内定义另一个函数，且这个函数能访问到外部函数的变量，即闭包是一个引用了另一个函数作用域中变量的函数
    应用场景：常用于回调函数、模拟块级作用域（立即执行函数）、用于数据的隐藏和封装，如创建私有变量和私有函数

  3. 解释原型链和继承

    • 每个对象都有一个关联对象，也就是原型对象，这个原型对象也有自己的原型，这些对象连接形成一条链条被称为原型链。当访问一个对象的属性和方法，如果没有，就会沿着原型链向上查找直到找到为止
    • 继承是一个对象能获得另一个对象的属性和方法，在JS中通过原型链实现，子类继承自父类的属性和方法

  4. this关键字的不同绑定规则（箭头函数、方法调用、构造函数等）

    • 全局上下文中this指向全局对象，浏览器为window，node中为global
    • 箭头函数没有自己的this，所以它的this就指向它所在上下文的this值
      • 在定义时绑定，继承自上下文
    • 方法调用的this指向当前调用该方法的对象
    • 使用new调用构造函数时，this绑定到新创建的对象上
    • 显示绑定，如call、apply、bind
      • call、apply会立即调用函数
      • bind会创建一个新函数实例，但不会立即调用
    • ES6类中的this绑定到新创建的实例上
    • DOM事件里this指向触发事件的元素
    • 隐式丢失，当从对象中引用方法时可能会丢失this，导致指向全局对象

  5. 解释异步编程（回调、Promise、async/await）

    • 异步编程是一种编程范式，可以在后台执行，使程序在等待某些操作（如文件读取、网络请求或定时器）完成时能够执行其他任务，从而提高效率和响应性
    • 同步是每个操作按顺序执行，如果某个操作需要等待，程序会阻塞，直到该操作完成
    • 实现异步编程的方式
      • 回调函数callback，通过将一个函数作为参数传递给另一个函数，当异步操作完成后调用该回调函数。该方式简单直观，但有多个异步操作顺序执行时，会出现嵌套过深，代码难以维护
      • promise是ES6引入的机制，用来传递异步操作的信息。避免了层层嵌套的回调函数，采用链式调用使代码更简洁。但中途无法取消，没有回调函数内部错误就无法得知
      • async/await是基于Promise的语法糖，async用于声明一个异步函数，await用于等待异步操作完成。代码更清晰、可读性更高。
        • 相较于promise，错误处理友好（可以用try/catch），调试友好（promise设置断点不友好）
    • 异步编程应用场景：网络请求、文件读写操作、定时器、用户输入或事件如点击、拖拽等
    • 异步执行机制依赖事件循环。
      • 宏任务：主代码块、setTimeout、setInterval、setImmediate(Node)、requestAnimationFrame()(浏览器)
      • 微任务：promise.then、catch、finally、process.nextTick(Node)
      • 微任务优先级高于宏任务，其中nextTick高于其他微任务，先进先出
      • 执行顺序是一个宏任务，所有微任务，GUI渲染，下一个宏任务

  6. 请解释一下事件冒泡、事件捕获、事件委托

    • 事件冒泡是事件被触发时，首先在目标元素上触发，然后逐级向上传播到文档根元素
    • 事件捕获与事件冒泡相反，事件从文档根元素开始向下传播到目标元素
    • 事件委托则是对大量相似的元素，将事件处理程序放到这组元素的共同祖先上，利用事件冒泡处理事件，通过event.target确定事件真正目标
    • 事件委托能减少事件处理器数量，提高性能；对于动态添加的元素也能触发事件，无需额外绑定

  8. 你能解释一下AJAX请求是什么吗？它和Fetch有什么区别？

    • ajax 是在不重新加载整个页面情况下能够与服务器交换数据并更新部分页面的技术
      • 特点：异步性，使用JS的 XMLHttpRequest 对象 与服务器通信，支持多种数据格式
      • 但配置和调用方式比较混乱，本身也是针对MVC编程
    • Fetch API是一个获取资源的接口，它基于Promise机制，简化了异步处理，用于替代 XMLHttpRequest
      • 支持 AbortController 和 AbortSignal 接口来中止请求
      • 只在网络故障或阻止请求时才报错，对响应为404、400、500会被当作成功的请求
      • 不默认携带cookie，需要手动设置credentials: ‘include’
      • 没有办法原生检测请求进度，而xhr可以
    • 两者区别：
      • fetch基于promise和原生js的封装，代码简洁；ajax基于回调函数，复杂异步逻辑时容易陷入回调地狱
      • fetch返回一个promise，直接可以链式调用；ajax需要手动处理readyState和status
      • fetch默认支持CORS，配置mode选项即可；ajax则需要服务端配合设置允许跨域
      • fetch支持读取流和处理大文件；ajax则不支持，需要将整个响应加载到内存中
      • fetch 更现代化，具有更强大的功能更灵活的配置；语法简洁
      • XMLHttpRuquest更具兼容性更成熟
    • Axios是基于Promise封装的HTTP客户端，在浏览器端会发起 XMLHttpRequest 请求，在node端发起 http 请求
      • 监听请求和返回，可以取消请求
      • 自动转化json数据；客户端可以抵御XSRF攻击

  9. Object.prototype.toString.call()、instanceof以及Array.isArray()这三者可以判断数组，有什么区别和各自的优缺点

    • Array.isArray()是ES5新增的方法，专门用于检测数组，不能判断其他类型

React

  1. 解释React的虚拟DOM及其工作原理

    • 用JS对象来描述DOM结构，实现高效UI更新的一种技术
    • 工作原理
      • 创建虚拟DOM树
      • 当组件状态或属性发生变化时，调用render生成新的虚拟DOM树
      • diff算法比较
        • react 将 新虚拟DOM树 与 旧的虚拟DOM树进行比较，这个过程称为 reconciliation（协调）
        • 从树根节点开始，比较同级节点，O(n)
        • 只比较同层级节点不跨层比较
        • 两个节点类型相同时，认为是相同的节点，只会比较和更新属性和子节点
        • 两个节点类型不同直接替换旧节点及其子节点
      • Patch操作应用到真实DOM，只更新变化的部分
    • 运用diff算法高效比较和更新

  2. 什么是React中的HOC（高阶组件）？

    • 是增强函数，接收组件作为参数，返回新组件
    • 实现代码复用，不改变原始组件结构或行为，无侵入性
    • 适用场景
      • 处理路由或组件的访问权限
      • 数据获取处理
      • 日志记录，错误处理
    • 组件臃肿
    • 使用 HOC 时要注意避免引入不必要的渲染性能问题，因为 HOC 会增加组件的层级
    • 确保 HOC 的参数化和配置化，使其具有更好的灵活性

  3. 解释React hooks的基本用法及其优势
    是react16.8推出的新功能，有了它能在函数组件里使用诸如state、各种生命周期等钩子函数的特性

    • 常用Hooks
      • useState，用于在函数组件中添加状态
      • useEffect，用于在函数组件中处理副作用
      • useContext，用于在函数组件中使用上下文
      • useReducer，用于在函数组件中管理复杂的状态逻辑
      • useCallback，返回一个memoized回调函数，用于性能优化
      • useRef，返回一个可变的ref对象，其.current属性被初始化为传递的参数
      • useLayoutEffect，类似于useEffect，但会在所有DOM变更后同步调用effect
    • 优点
      • 简化组件结构
      • 逻辑复用
      • 更少的代码
      • 性能优化，useMemo、useCallback
      • 利用函数组件和Hooks避免类组件中的 this 困扰

  4. 介绍React Fiber架构及其优势

    • 是react的新核心算法，旨在提高react性能和灵活性
    • 主要特征
      • 异步渲染，允许分割渲染工作，可以在多个帧中分批完成更新，从而避免阻塞主线程
      • 优先级调度，实现任务的优先级机制，确保高优先级任务可以打断低优先级的渲染任务，确保用户界面的响应速度
      • 并发模式，允许在后台进行非阻塞的渲染更新
      • 更好的错误处理，提供更细粒度的错误辩解，可以更有效地捕获和处理渲染过程中的国务
    • 优点
      • 提升用户体验，减少卡顿（异步渲染、优先级调度），快速响应（优先级调度）
      • 提高性能，分片更新，调度优化
      • 更灵活的渲染，并发模式支持，细粒度控制
      • 更好的开发体验，改进的错误边界，兼容旧版本

  5. 描述一下你对React的生命周期方法的理解，以及在何时会使用它们？

    • 挂载阶段
      • constructor()，组件构造函数，最先被执行。进行初始state设定和绑定事件处理函数
      • static getDerivedStateFromProps()，静态方法，会在组件实例化后和接收新的props前被调用
      • render()，渲染组件到DOM节点
      • componengDidMount()，组件挂载后立即被调用。进行API调用、数据请求、事件监听等操作
    • 更新阶段
      • static getDerivedStateFromProps(nextProps, prevState)，当组件接收到新的props时被调用
      • shouldComponentUpdate()(scu)，决定组件是否应该更新。进行性能优化，避免不必要的渲染
      • render()，组件更新时重新渲染
      • getSnapshotBeforeUpdate()，在DOM调用更新之前调用，可以获取DOM更新前的snapshot快照。在DOM更新前获取一些信息，如滚动位置
      • componengDidUpdate()，组件更新后立即被调用
    • 卸载阶段
      • componengWillUnMount()，组件卸载和销毁前被调用。清理工作，如清除定时器、事件监听
    • 错误阶段
      • static getDerivedStateFromError()，当渲染过程、生命周期方法或构造函数中抛出错误时，该方法被调用
      • componentDidCatch()，捕获子组件错误

  7. 函数式组编程怎么理解

    • 是一种编程范式，与面向对象编程相比，它强调无副作用的函数和不可变性
    • 数组的方法如split、join、map就有纯函数的特点
    • 核心概念
      • 纯函数，总是产生相同输出，不产生副作用
        • 例如函数不能包含new Date()或Math.random()等这种不可控的因素
      • 数据一旦被创建，就不可变；不修改参数
        • 保证数据稳定性，避免依赖的数据被未知地修改，引起自身执行异常
      • 高阶函数指接收一个或多个函数作为参数，返回一个函数的函数
      • 函数组合，将多个函数组合为一个新函数，可以一个接一个执行
      • 柯里化，将多个参数转换一系列只接收单个参数的函数
      • 递归
      • 鼓励将副作用隔离开（如修改全局变量、输入/输出等）
      • 不变数据结构，避免产生复杂的副作用
      • 惰性评估，当需要结果时才计算表达式的值
    • 优势
      • 完全独立，与外部解耦
      • 由于纯函数和不可变性，更易于理解和推理
      • 避免共享状态更适合并发和并行执行
      • 高阶函数和函数组合可以提高代码复用性
      • 纯函数没有副作用，更易编写单元测试
    • 缺点
      • 不可变性和递归可能导致性能问题，尤其是在处理大量数据时
      • 有一定学习成本

  1. Vue生命周期钩子有哪些，分别在什么时候触发？它们的作用是什么？
    beforeCreate，在实例初始化后，
    created，实例创建完成后，初始化注入和初始化打他、props等数据属性后，但未挂载到DOM上

核心思想

  1. 数据驱动视图：不直接操作DOM，而是通过修改数据来实现
  2. 组件化
  • 组件是独立和可复用的代码组件单元
  • vue组件可以使开发者使用小型、独立和通常可复用的组件构建大型应用
  • 提高应用开发效率、测试性、复用性等

Virtual DOM，是一种优化手段

是一个虚拟DOM树，用原生JS对象去描述一个DOM节点。借鉴了Snabbdom.js
Vue使用 Virtual DOM 来渲染组件，并且通过 diff算法 来比较 Virtual DOM 树的差异，从而减少操作 实际DOM

在Vue中，VNode是对 真实DOM 的抽象描述，有几个关键属性：标签名tag、数据data、子节点children、键值text、上下文环境context等
通过与真实 DOM 进行比较，可以找出需要更新的部分，从而最小化减少了 DOM 操作次数，提高了性能
diff算法复杂度o(n)

作用：

  • 维护视图和状态的关系，可以跟踪状态变化
  • 批量更新，最小化重绘重排
  • 复杂视图情况下提升渲染性能
  • 可以跨平台渲染

除了数据结构等定义外，要映射到 真实的DOM节点 上实际上还需要几个步骤：

  1. create 创建Virtual DOM树
  2. diff 比较新旧虚拟DOM
  3. patch 用于渲染对比VNode变成真实DOM

生命周期

  • 钩子函数 LIFECYCLE_HOOKS
  1. init初始化
    • 合并配置
    • 初始化生命周期、事件中心、渲染，往vm上挂载各种属性
    • 调用钩子函数 beforeCreate，此时实例刚创建
    • 初始化注入和初始化data、props、methods、computed、watch等属性（initState）
    • 创建完成，调用钩子函数 created，但此时还未生成真实DOM（添加顺序是先父后子）
    • 进行元素挂载，$el \ vm.$mount()
    • 如果没有render方法，compile编译得到render方法（其实就是template转为 字符串 返回给 render属性）
    • 渲染VNode前调用钩子函数 beforeMount
    • 执行render函数生成真实DOM，替换 DOM Tree
    • 调用钩子函数 mounted 组件已挂载（添加顺序是先子后父）
  2. 更新update
    • 实例化一个渲染 watcher并在 before() 中调用 beforeUpdate 钩子函数，监听数据变化并重新渲染
    • 在调用flushSchedulerQueue 函数时执行 update 钩子函数
  3. 组件激活、失活

      • 首次渲染、缓存渲染
  4. 组件销毁

数据响应，数据劫持

实现由两部分组成，分别是观察者(watcher) 和 依赖收集器(Dep)

  • Vue 2.0响应 是基于ES5 的 Object.defineProperty 实现，在对象上定义一个新属性或修改对象的现有属性并返回此对象
    • 设置 data 后，遍历所有属性，利用 defineProperty 重写 属性的 getter 与 setter 方法，从而完成监听数据的改变
    • 缺点：无法监听数组变化；无法处理属性也为对象的情况；深度监听需要递归到底，一次性计算量大；无法监听新增/删除属性
  • Vue 3.0响应 是基于ES6 的 Proxy 实现，new Proxy(target, handler)
    • Proxy 对象用于创建 一个对象的代理，从而实现基本操作的拦截和自定义操作，如属性查找、赋值、枚举、函数调用等
      • 应用场景：校验器、私有属性…
    • 优点：不需要深度遍历监听，并且对象、数组变化都能监听得到(能观察的类型更丰富)
      • 但不兼容IE，也没有 polyfill, defineProperty 能支持到IE9
  • 两者都是对 对象上的某个属性 做处理
    • Object.definedProperty 是劫持 对象的属性，新增元素需要再次 definedProperty
    • Proxy 劫持的是 整个对象，不需要做特殊处理，用到某个属性时才去收集依赖

响应式原理

基于 数据劫持 和 发布-订阅模式，当数据发生变化时，会触发相应的更新操作，从而更新视图

  • 观察者模式

    • 被观察者 subject、观察者 observe
    • 是一对多的关系，是耦合的，适合 在组件内部操作

  • 发布-订阅模式

    • 订阅者 subscriber、发布者 publisher
    • 增加一个消息中心 Dep，接到指令后会根据事件类型给对应的订阅者发送消息
    • 添加消息中心，所有消息均通过消息中心，发布者与订阅者不直接相关
    • 实现了两者解耦，更适合 消息类型复杂的情况

回流必定触发重绘，反之不一定
重绘开销小，回流代价大

  • javascript方面

    • 减少DOM操作，将多次合并为一次操作；缓存DOM引用避免在循环中频繁查询DOM
    • 尽量使用class进行样式修改而不是内联样式减少重排次数
    • 避免触发同步布局如offsetWidth、scrollTop等引发同步计算导致频繁重排（读写分离，先取值element.offsetWidth再赋给style.width）

    • 避免写出深层嵌套和复杂选择器的css

在浏览器中输入URL到展示的过程

可以大致描述为：
输入URL - 发起URL请求（DNS解析、建立TCP连接、发送请求、服务端处理请求并发送响应）- 浏览器接收响应 - 渲染页面

这一过程共涉及到3个进程，分别是浏览器进程、网络进程和渲染进程

  1. 处理输入，当在地址栏输入内容时，地址栏先判断是搜索查询还是请求URL，如果是搜索内容，会发送到搜索引擎合成新的带搜索关键字的URL。如果是域名会加上协议（如https）合成完整的url。
  2. 点击回车后，浏览器进程通过IPC把URL传给网络进程，网络进程获取到URL先去找缓存中是否有缓存资源，如果有就拦截请求直接返回资源。没有则请求DNS返回域名对应的IP地址和端口号。
  3. 建立TCP连接，连接建立后浏览器端构建请求行、请求头、请求正文等数据包，向服务器发送请求
  4. 服务端收到请求信息后处理构建响应数据发送回浏览器的网络进程（这一步后断开TCP连接）
  5. 网络进程接收到响应数据后进行解析，比如检查状态码、响应处理（重定向在这一步发生，获取URL重新来过）
  6. UI线程通知渲染进程开始渲染页面，渲染过程：
    1. 解析HTML转换为DOM树（就算编写错误也会被HTML规范优雅地处理掉）
    2. 加载子资源，包括图像、CSS和JS等外部资源
      • script标签会暂停HTML文档的解析，必须加载、解析和执行JS代码后再恢复HTMLz文档的解析
    3. 解析CSS创建CSSOM树（CSS对象模型）并确定每个dom节点的计算样式
    4. 主线程遍历DOM树和CSSOM树创建布局树
    5. 主线程遍历布局树来创建绘制记录和图层树（绘制记录有点像canvas规定了先画什么）
    6. 确定绘制顺序后，开始合成显示在屏幕上
  7. 提交导航，更新浏览器界面状态，包括地址栏、浏览器历史记录、网络安全状态等
  • beforeunload方法
    • 如果导航是从渲染器进程启动的，比如用户单击链接或JS运行window.location，则渲染器进程首先检查beforeunload处理程序。然后进行与浏览器进程启动导航相同的步骤。
  • TCP建立连接，客户端和服务端通过3次握手来确保双方能正确识别对方序列号，从而实现可靠的通信
    • 客户端发送SYN（同步序列号）包，告诉服务器希望建立连接并指定了一个初识序列号
    • 服务端确认客户端序列号，发送SYN-ACK包
    • 客户端接收上述包后，发送ack包确认服务器序列号也发送自己的序列号，表示连接已建立
  • 关闭TCP连接，通过4次挥手确保连接正常终止，避免数据丢失或不完整的情况
    • 客户端发送FIN包，表示完成数据发送并不再需要连接（这个包中也包含了一个序列号）
    • 服务端回复ACK包确认收到客户端的FIN包
    • 服务端发送FIN包表示准备好关闭连接了（同样也包含了一个序列号）
    • 客户端回复ACK包确认了服务器序列号，表示接受到关闭连接的请求
  • 如果导航为重定向跨站点，则可能不会使用此备用进程，在这种情况下可能需要不同的进程
  • DNS是域名系统，互联网中将域名转换为IP地址的分布式数据库系统

是一种用于传输超媒体文档的协议，在客户端和服务器之间作为请求-响应协议工作

  • HTTP不安全，以纯文本传输数据
  • HTTP/1.0缺陷
  • HTTP/1.1改进
  • HTTP/2、HTTP/3
  • HTTPS超文本传输安全协议，HyperText Transfer Protocol Secure*
  • 使用SSL/TLS加密数据 双方使用随机密钥来加密和解密通信过程中的数据
    用于保护网站敏感数据，如用户登录、支付信息等

安全性

  • XSS攻击，跨站脚本攻击
    • 目标是向网页中注入恶意脚本代码，从而在用户浏览页面时执行这些恶意代码
    • 通常分为三种：
      1. 存储型XSS攻击，恶意脚本代码存储在网站数据库中，当访问页面时，会从服务端返回并执行
      2. 反射型XSS攻击，将恶意链接发送给用户，用户点击后返回恶意代码并执行
      3. DOM型XSS攻击，构造特定的URL或其他用户可控制的输入，当访问页面时，代码执行
    • 防御措施：
      • cookie设置httpOnly
      • 对用户输入进行严格过滤，转译页面上的输入内容和输出内容
      • 限制JavaScript执行的权限
  • CSRF(Cross-Site Request Forgery)跨站请求伪造
    • 在登录过的网站上进行非预期无感知的操作，比如向目标网站发送恶意请求，从而可以发起类似资金转账、更改账号信息等未授权的操作
    • 防止攻击的常见方法：
      1. 使用CSRF Token，随机生成Token，并在服务端检验有效性，确保请求合法
      2. 同源检测
      3. 使用 SameSite Cookie属性，限制Cookie发送，防止跨站点请求
      4. 设置合适的HTTP头部（X-Ruquested-With、Content-Type…）确保请求安全性
      5. 使用双重身份验证，增加攻击者难度
      6. 设置白名单，不被第三方网站请求；不被第三方网站访问到用户的cookie

存储机制

一般主流方案：Cookie、Web Storage、IndexedDB、Cache Storage

Cookie，通过浏览器在客户端存储少量数据

  • 存储量较小，通常为4kb大小；
  • 可以被浏览器设置，每次请求都会自动携带，包含在请求头上，不安全
  • 也可以被服务器设置
  • 使用场景比较有限，通常用于存储用户信息，登录状态等

Web Storage，在客户端保存键值对数据，限制为4～5MB

  • LocalStorage，在浏览器关闭后仍然保存，直到手动删除。使用方便
  • SessionStorage，在当前会话结束时即被清除

IndexedDB，一种高级的客户端存储技术，允许在浏览器中存储大量结构化数据并支持查询

  • 存储容量大，支持事务，可高效地处理大量数据
  • API相对复杂，不同浏览器兼容性不同

Cache Storage，浏览器提供的一种缓存机制

  • 用于存储请求过的资源，如CSS、JS、图片等
  • 提高页面加载速度，减少网络请求，提升用户体验

性能优化

可以考虑采取以下措施提高性能：

  • 使用局部变量来提高访问速度
  • 避免使用eval函数，因为它会影响性能
  • 使用事件委托来减少事件处理程序的数量
  • 减少DOM操作次数，避免频繁的重绘和重排；可使用批量更新
  • 使用合适的数据结构和算法
  • 使用异步加载来延迟不必要的资源加载
  • 缓存重复的计算结果
  • 使用Web Workers进行多线程处理，提高性能

跨域

是浏览器基于同源策略的一种安全手段，是对安全性做的限制

同源策略：是一种约定，用于隔离潜在恶意文件的安全机制，通常不允许不同源之间的读操作

同源三要素：协议protocol / 主机host / 端口port 相同
非同源的请求就会产生跨域

常见于以下场景：

  • ajax跨域请求
  • 跨文档消息通信，一个页面通过postMessage发送消息到另一个页面时，如果两个页面的域名不同，则会触发跨域问题
  • 嵌套的跨域资源，如iframe、img、script
    解决方案：
  1. JSONP（JSON with Padding）动态创建script标签来加载脚本，实现跨域数据传递
    • 可兼容老式浏览器
    • 需要服务端支持且只支持GET请求
  2. CORS（Cross-Origin Resource Sharing）跨境资源共享。服务端设置响应头Access-Control-Allow-Origin: * | <origin>，允许指定的域名访问资源
    • 支持所有请求方法，但兼容性比JSONP差
    • 实现关键在服务端
  3. 代理服务器，在服务端设置代理服务器，由代理服务器转发请求，实现跨域访问。（node中间件）
  4. 反向代理，将需要访问的资源部署在同一域名下，通过反向代理将请求转发到真实的资源地址。（Nginx）
    • node中间件和Nginx反向代理都利用了服务器对服务器无同源策略
  5. iframe嵌套通信，document.domain + iframe / window.name + iframe / location.hash + iframe
  6. 基于OPTIONS方法进行过滤，发送一个预检请求到服务器，是否允许其实际请求
  7. postMessage，是HTML5 XMLHttpRequest Level 2中的API，可以实现跨文档通信
  8. webSocket，是HTML5的一个持久化协议，不受同源策略限制，可以建立跨域连接，可与postMessage配合使用。是一个长连接long poll

代理服务器 与 反向代理服务器 区别：

  1. 代理服务器位于客户端和原始服务端间，而反向代理服务器位于客户端和多个原始服务器间
  2. 代理服务器主要用于隐藏客户端的真实IP地址、提高访问速度、过滤内容等
  3. 反向代理服务器主要用于负载均衡、安全性、隐藏服务器架构等方面
  4. 代理服务器的配置由客户端控制，需手动配置代理服务器的地址和端口
  5. 反向代理服务器的配置由服务端控制，客户端无需感知反向代理服务器的存在

实际应用中，反向代理服务器更为常见。
特别是在构建高可用性、高性能的Web应用时
通过反向代理，可以将请求分发给不同的服务器，实现负载均衡和故障转移；
同时，反向代理也可以用于缓存静态内容，提高访问速度；
此外，反向代理还可以作为防火墙，过滤恶意请求，增强系统的安全性。

Node的Event Loop事件循环

Timers -> Pending callbacks -> Idle/Prepare -> IO Poll -> Check -> Closing callbacks

  • Timers 用于存储定时器的回调函数(setInterval、setTimeout)

  • Pending callbacks 执行与操作系统相关的回调函数，比如启动服务器端应用监听端口操作

  • Idle/Prepare 系统内部使用，可以不关注

  • IO Poll 存储I/O操作的回调函数队列，例如文件读写操作

    • 如果事件队列中无回调函数，但 Check阶段 或 Timers队列 中有需执行的回调函数
      • 那么事件循环会移至 Check阶段 或 Closing callbacks 并从 Timers 进入下一次循环
      • 否则会在IO Poll阶段停留一段时间以等待新的回调函数进入

  • Check 存储setImmediate API 的回调函数

  • Closing callbacks 执行与关闭事件相关回调函数，比如关闭数据库连接

  • Node适合I/O密集型任务，不适合CPU密集型任务，因为主线程一旦阻塞，就会卡程序

  • 宏任务：setInterval、setTimeout、I/O、setImmediate

  • 微任务：Promist.than/catch/finally、process.nextTick

  • 宏任务微任务各自的回调函数放在各自的任务队列中

  • 微任务优先级高于宏任务

  • 微任务中的nextTick高于其他微任务，先进先出

HTTP请求方法

get / post / options / put / patch / delete / head / trace

  • get，用于向服务器请求获取数据，如获取网页、图片等
    • 请求参数附在URL上，以查询字符串形式发送
    • 请求长度受限，不修改资源，无副作用
    • 可被缓存，也可以被书签、历史记录保存
    • 数据在URL中可见，不适合发送敏感信息
  • post，用于向服务器提交数据，如表单数据、上传文件等操作；更多编码格式
    • 请求参数附在请求体中，不会在浏览器历史记录中显示
    • 不可被浏览器缓存，不会被书签、历史记录保存
    • 数据不暴露在URL中，适合发送大量数据和敏感信息
  • head，类似于get，但服务器只返回请求头部信息，通常用于检查资源的元数据，例如文件是否存在、最后修改时间
  • options，用于请求服务器告知客户端对指定资源支持的各种请求方法、HTTP头部信息等
  • put，用于向服务器发送数据，从而更新指定资源内容。如果资源不存在，则创建一个新资源
  • patch，类似于put，但仅传输需要更新的数据部分，非整个资源
  • delete，用于请求服务器删除指定的资源
  • trace，用于在请求链路中追踪请求的传输路径，常用于诊断和调试

进程、线程

  • 进程是CPU资源分配等最小单位，是应用程序运行时的实例对象；进程间相互独立不受干扰；可以通过IPC来实现通信
  • 线程是CPU调度等最小单位，被包裹在进程中，应用程序具体要做的每件事情生成了一个个线程，类似于一个待办列表
  • 线程上下文切换比进程上下文切换要快得多
  • CPU可以有很多进程，但资源是固定的。进程越多，CPU分出去的资源越多就越卡。

JS为什么是单线程

这是由起初设计的用途和运行环境所决定

  • 起初被设计用于web页面互动及操作DOM，如果是多线程，会带来复杂的并发问题，如死锁（两个线程相互依赖）、同步问题、竞态条件（多个线程访问共享资源引起的不确定性和不可预测的行为）等
  • 多个js脚本同时修改页面可能会引起页面冲突，单线程保证了脚本顺序执行
  • 在每个页面中多线程可能会导致页面响应变慢或崩溃
  • 可以通过事件循环机制，处理异步任务，如网络请求、定时器等

web worker，是HTMl5提供的JS线程

  1. 独立线程，与主线程相互独立，不阻塞主线程。可并行执行多个任务
  2. 无法访问和操作DOM元素
  3. 支持多线程通信
  4. 受同源限制，无法加载本地资源
    使用场景：需要大量计算、网络请求等耗时的任务

事件冒泡、事件捕获、事件委托

  • 事件冒泡，当一个事件在DOM元素上触发时，它会首先在事件的目标元素上触发，然后逐级向上传播到文档的根元素。这个过程就像泡泡一样从底部浮到顶部
    • 可调用event.stopPropagation() 来阻止事件继续冒泡
  • 事件捕获与冒泡相反，从文档根元素开始，向下传播到目标元素
    • 可调用event.stopPropagation() 来阻止事件继续向下传播到目标元素
  • 事件委托对大量相似的元素，将事件处理程序添加到这一组元素的共同祖先上
    • 利用 事件冒泡 来处理事件，通过检测event.target属性确定事件的真正目标
    • 减少事件处理器的数量，提高性能（在处理大量相似元素上）
    • 对于动态添加的元素也可以触发事件，无需额外绑定
    • 简化代码，易于管理

盒模型

页面渲染时dom元素所采用的布局模型，是CSS布局的基础。
它规定了元素如何计算其大小和位置，它将所有元素表示为一个个矩形盒子。
每个盒子由四部分组成:

  • 内容区域content area / 内边距区域padding area / 边框区域border area / 外边距区域margin area
    可通过box-sizing设置，根据计算页面宽高的区域可分为：
    • content-box (w3c 标准盒模型)
    • border-box (IE 盒模型)
    • padding-box
    • margin-box

默认为content-box，即w3c标准盒模型

box-sizing属性用于更改默认的CSS盒模型计算方式

每个盒子有四个边界：

  • 内容边界Content edge / 内边距边界 Padding Edge / 边框边界 Border Edge / 外边框边界 Margin Edge

IE盒模型缺陷，指早期版本的IE调整网页元素大小的方法和W3C推荐的标准方式不同。在IE6中浏览器支持了一种解决这种差异的可选渲染模式（遵从标准模式）

BFC (block formatting context)，块级格式化上下文

  • web页面可视css渲染的一部分（css布局的一个概念）
  • 块级盒子布局过程发生的区域
  • 浮动元素与其他元素交互的区域

下列方式会创建BFC或触发条件:

  • html根元素
  • 浮动元素, 即float不为none的元素
  • 绝对定位元素, position: absolute / fixed
  • 行内块元素, display: inline-block
  • 表格单元格, display: table-cell
  • 表格标题, display: table-caption
  • display: flow-root的元素
  • contain值为layout / content / paint
  • overflow不为visible / clip的块元素
  • 弹性元素, display: flex / inline-flex的直接子元素
  • 网格元素, display: grid / inline-grid的直接子元素
  • 匿名表格单元格元素, display: table / table-row / table-row-group / table-header-group / table-footer-group / inline-table
  • 多列容器, 元素的column-count或column-width不为auto, 包括column-count: 1
  • column-span为all的元素始终会创建一个新的BFC, 即使该元素没有包裹在一个多列容器中

BFC有哪些作用:

  • 自适应两栏布局 (display: flow-root)

  • 可以阻止元素被浮动元素覆盖(清除外部浮动)

  • 包含浮动元素–清除内部浮动（两个div都位于同一个BFC之中）

  • 可以阻止margin重叠（当分属于不同BFC时）

    BFC对浮动定位与清除浮动都很重要。
    来自：两个块元素外边距合并问题

层叠上下文

HTML元素沿着面向用户的一条虚构的z轴顺序排开，是对HTML元素的一个三维构想
运用z-index，将元素提升为一个比较特殊的图层，在三维空间(z轴)中高出普通元素一等

  • z-index属性设置元素的堆叠顺序
  • 拥有更高堆叠顺序的元素总是会处于顺序较低的元素的前面
  • z-index仅能在定位元素上奏效（position属性值不为static以外的元素，包括relative，absolute，fixed样式）

满足以下任意一个条件的元素就会形成层叠上下文:

  • html根元素
  • position为absolute / relative且z-index不为auto
  • position为fixed / sticky
  • 网格布局grid或弹性布局flex容器里的子元素且z-index不为auto
  • opacity属性值小于1的元素
  • …
  • 层叠上下文可以包含在其他层叠上下文中，并且一起创建一个层叠上下文的层级
  • 每个层叠上下文都完全独立于它的兄弟元素（当处理层叠时只考虑子元素）
  • 每个层叠上下文都是自包含的（当一个元素的内容发生层叠后，该元素将被作为整体在父级层叠上下文中按顺序进行层叠）

居中布局

  • 水平居中
    • 行内元素：text-align: center;
    • 块级元素： margin: 0 auto;
    • absolute + transform
    • flex + justify-content
  • 垂直居中
    • 上级高度可知时：line-height: height
    • absolute + transform
    • flex + align-items
    • table布局
  • 水平垂直居中
    • absolute + transform
    • display: flex + justify-content + align-items
    • display: grid + justify-content + align-items
    • display: table-cell + text-align + vertical-align

positon 该属性用于指定一个元素在文档中的定位方式

  • static：正常布局，即元素在文档常规流中当前的布局位置。top、left、right、bottom、z-index无效，按正常文档流进行排列
  • relative：元素不脱离文档流，在原始位置上不改变页面布局的前提下调整元素位置，不影响其他元素偏移。未添加定位时的位置会多出一块空白
  • sticky：元素根据正常文档流进行定位，然后相对他的最近的滚动祖先和块级祖先进行偏移，偏移值不影响任何其他元素位置
  • absolute：元素会被移出正常文档流，不预留空间，可设置外边距，不与其他边距合并
  • fixed：元素会被移出正常文档流，不预留空间，通过指定元素相对于屏幕视口的位置来指定元素位置，位置随屏幕滚动时不会改动，所固定参照对象是可视窗口
  • absolute的containing block计算方式跟正常流有什么不同？
    • 正常流中，元素的containing block是其最近的块级祖先元素
    • 但绝对定位absolute可能会是以下几种：
      • position: relative / absolute的祖先元素
      • 离它最近的包含块containing block，即最近的块级祖先元素

V8，Google开发的JS引擎

负责JS代码编译和执行
具有高效的即时编译、优化编译和垃圾回收机制，特点是高性能
V8将堆内存分为：

  • 新生代，存放生命周期较短的对象，采用Scavenge算法垃圾回收，会频繁地对新生代进行清理
  • 老生代，存放生命周期较长的对象，采用标记清除和标记整理等算法垃圾回收

垃圾回收机制，负责管理和释放不再被引用的内存

  1. Scavenge算法
    • 采用复制算法，将存活的对象从一个空间复制到另一个空间
    • 将内存空间一分为二，分为From空间 跟 To空间；From空间存放活对象
    • 当From空间被占满时，启动垃圾回收，将存活的对象复制到To空间，进行内存整理，最后From空间和To空间互换
  2. 标记清除，主流垃圾收集算法
    • 给存储在内存中的所有变量加上标记
    • 去除环境中及被环境中的变量所引用的变量的标记
    • 剩下标记的变量就被视为准备删除的变量

会产生内存碎片，导致内存空间不连续性

  1. 标记整理，标记清除改进版

    • 清理不再被引用对象之后，会对存活的对象进行整理，移动到内存的一端，减少内存碎片产生
    • 移动对象需要更新所有引用这些对象的指针，开销较大

  2. 引用计数，不太常见

  • 如果一个值的引用次数为0时，就会被释放内存
  • 函数被反复多次调用，就会导致大量内存得不到回收
  • 循环引用会导致问题，可能会导致内存永远不会被回收（导致引用次数始终大于0）
  1. 三色标记算法
    • 使用白灰黑三种颜色标记对象状态
    • 能减少停顿时间，可以与程序执行并发进行（有新的引用也能及时观察到）

  • 为执行中的代码只保存必要的数据
  • 尽可能少未声明就初始化变量，会变为全局变量
  • 解除引用：不仅可以消除循环引用，对垃圾回收也有帮助
    • 设置为null 来释放其引用，让值脱离执行环境
    • 局部变量会在 离开执行环境时 自动被解除引用（函数执行完毕，局部变量就会被销毁）
  • JS中内存主要有V8引擎负责
    • 栈内存用于存储原始类型和函数调用帧
    • 堆内存则用于存储对象和函数

性能问题

  • let、const都以块为作用域，相比于使用var可能会更早地让垃圾回收程序介入，尽早释放内存
  • 隐藏类和内嵌缓存来提高性能
  • 尽量做到及时回收不用的内存

内存泄漏

一些常见的内存泄漏情况：

  1. 意外的全局变量，在页面声明周期内不会被释放导致无法回收
  2. 定时器未清理会导致函数或对象一直存在于内存中无法被回收
  3. 事件监听器未被正确移除，仍然存在于内存中
  4. 如果闭包中引用了大量变量而又未被及时释放
  5. dom引用：如果在不需要时未释放这些引用，导致dom元素无法被回收
  6. 循环引用无法被垃圾回收
  7. 在内存中缓存大量数据未及时清理

怎么避免呢？

针对上述的情况，可以多注意管理事件监听器、定时器等，及时释放不需要的资源

  • 合理使用变量和对象
  • 避免循环引用
  • 优化算法和数据结构
  • 及时释放资源，如打开文件、网络请求等，务必在使用完毕后及时关闭或释放资源
    还可以通过工具来监控和诊断问题，比如Google浏览器有一些工具

    • void 0

    • Number.MAX_VALUE / .MIN_VALUE
      • 超出范围的值将自动转换为Infinity / -Infinity
      • 用isFinite(..)判断是否位于最大和最小数值之间
    • Number.MAX_SAFE_INTEGER，最大安全数字，这个数范围内不会出现精度丢失，超出就会出现计算不准确的情况
  • 浮点数
    • 由于浮点数值需要的内存空间是保存整数值的2倍
      • 像1.这样的小数点后无数字，那么会被作为整数值来保存
      • 最高精度为17位小数，计算会产生舍入误差问题；toPrecision(..)

  • NaN，Not a Number
    • 涉及NaN的操作都返回 NaN
    • 与任何值都不想等，包括NaN本身
    • 当对对象调用isNaN(..)时
      • 先调用对象的valueOf(..)，确定返回值可否转换为数值

      • parseInt(string, radix)
        • radix 是2-36之间的整数，表示被解析字符串的基数
    • Number(..)函数基于一些规则执行转换，其中null返回 0；undefined返回 NaN

    • 字面量 \n \t \b \r \f \ ' " \xnn \unnnn

闭包

一个引用了另一个函数作用域中变量的函数
常见方式：在一个函数内部创建另一个函数
是一种静态作用域

  • 闭包作用域链包含自己的、包含函数的和全局的
    • 在作用域链中查找变量 (作用域链本质上是一个指向变量对象的指针列表)
  • 一般来讲，函数执行完毕后，局部活动对象就会被销毁，内存中仅保存全局作用域
  • 而闭包则会使内部定义的函数将包含函数的活动对象添加到它的作用域链中
    • 当被引用的函数返回后，其执行环境的作用域链会被销毁，但活动对象仍会留在内存中
  • 闭包只能取得包含函数中任何变量的最后一个值，因为它们引用的是一个活动对象里的变量
    • 闭包保存的是整个变量对象，而不是某个特殊变量
  • 匿名函数的执行环境具有全局性，this通常指向window
    • 要想访问作用域中的arguments对象，必须将该对象的引用保存到另一个闭包能访问的变量中
  • 闭包还可以用于在对象中创建私有变量、私有函数
    • 有权访问私有变量的共有方法叫 特权方法
      • 单例模式、构造函数模式、原型模式
    • 在原型prototype上定义公有方法 例如get() set()
      • 但在一个实例上调用时会影响所有实例，可能会导致返回相同的值
  • 数据的隐藏和封装，用于创建私有变量和方法
  • 模拟块级作用域（立即执行函数）
  • 常用于回调函数
  • 闭包会引用包含函数的整个活动对象，有可能会造成引用无法解除，占用内存
    • 创建闭包必须要维护额外的作用域，过度使用可能会占用大量内存

解决：

  • 变量可以通过函数参数的形式传入，避免使用默认的[[scope]]向上查找
  • 使用setTimeout包裹，通过第三个参数传入
  • 使用块级作用域，让变量成为自己上下文的属性，避免共享
  • 避免循环引用，确保不再需要时将外部函数的变量引用置为null

执行环境 全局执行环境 函数执行环境

  • 通常指代码执行时的整个环境，包括当前作用域、变量、函数等信息
  • 全局执行环境为 window对象
  • 每进入一个新的执行环境，都会创建一个用于搜索变量和函数的作用域链
  • 当代码在一个环境中执行时，会创建变量对象的一个作用域链，保证对执行环境有权访问的所有变量和函数的有序访问
    • 哪些方式能延长作用域链
      • 嵌套函数、闭包函数（在内部访问外部环境的变量）
      • try-catch中的catch块（将变量赋值给外部作用域的变量来延长作用域）
      • with语句
      • ES6模块（使变量和函数在不同模块间共享和使用）

执行上下文EC

可以简单理解为一个对象，可以认为是 执行环境的一部分，更偏向于描述代码执行时的特定上下文信息。
包含三部分：

  1. 变量对象VO，存储该执行上下文中所有变量和函数声明
    • 当变量对象所在的上下文为active EC时，称为活动对象AO
  2. 作用域链，本质是一个指向变量对象的指针列表，包含父级的[[scope]]和自己的变量对象AO
    • 作用域scope是变量和函数可访问的区域或上下文。可以分为块级作用域、函数作用域
    • 在JS中 作用域 是通过 作用域链 连接起来的
  3. this指向
    • 类型：全局执行上下文caller / 函数执行上下文callee / eval执行上下文
  • 变量或函数的上下文 决定了它们可以访问哪些数据以及行为
  • 每个上下文都有一个关联的变量对象
  • 上下文中的代码在执行时，会创建变量对象的作用域链
  • 上下文之间连接是线性的、有序的

Function

  • 每个函数都是 Function 的实例
  • 函数是对象，函数名是指向函数的指针，所以函数可以用作 值 来使用

  • 解析器会率先读取 函数声明，function xxx(..){..} 声明提升了
  • 函数表达式则在 解析器执行到它所在行时，才会被 解释执行 var xxx = function(){}
  • 函数内部有两个特殊对象：
    • arguments：类数组对象，保存函数参数
    • callee：指向拥有这个arguments对象的函数（函数执行上下文），典型的有 阶乘函数
  • this，引用的是函数执时行的环境对象
  • caller，保存着当前函数的函数引用 arguments.callee.caller
  • 函数属性：
    • length：表示函数希望接受的命名参数的个数
    • prototype：保存所有实例方法
  • 每个函数都包含两个非继承而来的方法：能扩充函数赖以运行的作用域，使对象不需要与方法有任何耦合关系
    • apply (作用域, 参数数组arguments/array)

原型

用于实现对象的属性继承，行为重用
通过复制已存在的原型对象来实现，但这里的复制并不是真实的复制，而是使新对象持有一个原型的引用
简单来说有两点：

  1. 每个对象有私有字段prototype，即对象原型
  2. 读取属性时，如果对象本身没有，就向上查找原型直到找到为止
  • 每个函数都有一个 prototype 属性，该属性是一个指针对象，是一个对象
  • Object内建对象 是 最高级父级对象
  • 该对象包含由 特定引用类型的实例 共享的属性和方法
  • 好处是可以让 所有对象实例共享它所包含的属性和方法
  • 所有原型对象会自动获得一个 constructor（构造函数）属性，指向与之关联的构造函数
  • 当为对象实例添加一个属性时，就会 屏蔽原型对象中保存的 同名属性，即会阻止访问原型中的属性，但不会修改那个属性
    • 自身属性优先级高于其原型对象中同名的属性
  • Object.getPrototypeOf(..) / for in / Object.keys(..) / Object.getOwnPropertyNames(..) / isPrototype(..)
    • 使用hasOwnProperty(..)判断哪个是实例属性，哪个是原型属性；只有属性存在于实例中时才返回true
    • for in (“xxx” in p1) 只要能访问到就为true
      • 实例和原型中的属性都包括（能通过对象访问的、可枚举的属性）
    • Object.keys()返回包含所有可枚举属性的字符串数组
    • Object.getOwnPropertyNames()无论是否枚举都返回
  • 使用 对象字面量形式 创建时，相当于重写了prototype对象，constructor属性就会变成新的，指向了Object构造函数

原型陷阱: 当重写了整个原型对象时，就切断了跟先前已经创建的实例对象之间的联系，已经创建的实例引用的依然是最初的原型

  • 在原型中查找值的过程是一次搜索，对原型对象做的任何修改都能够立即从实例上反映出来
  • 当原型对象中包含引用类型值属性时，当通过某个实例改变值时会导致所有实例指向的该属性值都会改变
  • 推荐使用构造函数 + 原型模式组合使用
  • 寄生构造函数模式，稳妥构造函数模式（安全性）

继承

继承是一个对象获得另一个对象的属性和方法的能力，JS中继承通过原型链实现。

原型链

由原型对象组成，每个对象都有__proto__属性，指向创建该对象的构造函数的原型，将对象连接组成原型链。
是一个用来实现继承和属性共享的对象链
通过原型链继承原型上的属性或方法

  • __proto__是某个实例对象的(非标准)属性，prototype则是属于构造器函数的属性
  • 所有函数的默认原型是Object的实例，默认原型的指针指向Object.prototype
    • 即所有自定义类型都会继承toString()、valueOf()等默认方法
  • 用instanceof(..), isPrototypeOf(..)确定原型和实例的关系
  • 实现继承时，不能使用对象字面量创建原型，这样就重写原型链，切断了相互间的关系
  • 在创建子类型实例时，不能向超类型的构造函数中传递参数，解决方法如下：
    • 借用构造函数，在子类型构造函数内部调用超类型构造函数
      • xxx.call(this，参数值)
  • 实例通过 __proto__ 指向 原型prototype，通过 constructor 指向构造函数。

组合继承

  • 使用原型链实现对原型属性和方法的继承
  • 通过借用构造函数来实现对实例属性的继承
  • 这样既在原型上定义方法实现函数复用，又能保证每个实例都有它自己的属性

原型式继承

  • 借助已有对象创建新对象，实现对某个对象的简单继承
  • 如ES5的 Object.create()

寄生式继承

  • 借用构造函数来继承属性，通过原型链等混成形式来继承

  • ES5中 Object.create(person, object) 是静态方法，以一个现有对象作为原型，创建一个新对象

    • 规范了原型是继承，比用 new 来创建对象更规范些
    • object { xxx: { vlaue: ‘’} }

  • ES5继承是先创建子类实例对象this，再将父类方法添加到this上(Parent.apply(this))（实例在前，继承在后）

  • ES6语法糖 class / extends 也可以实现继承

  • ES6继承是先创建父类实例对象this，再用子类构造函数去修改this(super(…args))

    • 只有调用super后才能使用this
    • 继承在前，实例在后

JavaScript 引擎会在 编译阶段 进行数项的性能优化

这些代码会如何对作用域进行修改，也无法知道传递给 with 用来创建新词法作用域的对象的内容到底是什么。

  • 最悲观的情况是如果出现了 eval()或with()，所有的优化可能都是无意义的，因此最简单的做法就是完全不做任何优化。
  • 使用任何一个机制都将导致代码运行变慢

调用栈

  • 为了到达 当前执行位置 所调用的所有函数，可以当作一个函数调用链
  • 非严格模式下默认绑定会绑定到 全局对象
  • 调用位置是否有上下文 a.b.c，隐式绑定会把函数c中调用的this绑定到这个上下文对象中，对象属性引用链中只有最后一层（即b）影响调用位置
    • 隐式绑定到函数会丢失绑定对象，就会引用 默认绑定
  • 显式绑定
    • call(..) | apply(..) | bind(..) (Function.prototype.bind）硬绑定
    • 把 null 或 undefined 作为this的绑定对象传入 call ｜ apply ｜ bind，这些在调用时会被忽略，实际应用默认绑定规则
  • new 创建的函数，this绑定的是新创建的对象
  • 优先级： new > 显式绑定 > 隐式绑定
  • 箭头函数的绑定无法被修改
  • JS中实际上不存在所谓的构造函数，只有对于函数的构造调用
    • 创建一个全新对象
    • 新对象会被执行 原型连接
    • 新对象会绑定到函数调用的this
    • 如果函数没有返回其他对象，那么new表达式的函数调用会自动返回这个新对象
  • Object.create(..) 会创建一个新对象并把它关联到我们指定的对象
    • 通过 Prototype机制 委托来关联对象并且避免不必要的麻烦
    • 使用 new 的构造函数调用会生成 .prototype 和 .constructor 引用
  • Object.create(null) 和 {} 很像，但是并不会创建 Object.prototype 这个委托
    • Object.create(null)没有原型链，创建的对象没有原型，所以它比 {} ‘更空’
    • 这类对象通常被称为 纯净对象 或 字典对象

尾调用优化

  • 尾调用就是一个出现在另一个函数“结尾”处的函数调用（某个函数最后一步是调用另一个函数）
  • 函数调用会形成一个调用记录（调用帧），所有的形成一个调用栈
    • 当出现尾调用时，如函数A内最后调用了B，此时A就结束了，那完全可以删除A的调用帧，只保留B的调用帧即可
    • 每次执行时调用帧只有一项，将大大节省内存
    • 但只有不再用到外层函数的内部变量时，才能进行尾调用优化（闭包就不行）
  • 在递归问题上可以更好地运用起来，即尾调用自身，只保留一个调用记录，复杂度为O(1)
  • ES6中尾递归不会发生栈溢出，相对节省内存
    • ES6的尾调用只在严格模式下生效

模块

简单来说就是将代码分成独立的块，再把这些块连接起来
思路：逻辑分块、各自封装、相互独立，每个块自行决定对外暴露什么，同时也自行决定引入哪些外部代码。

模块必须在依赖加载完成后才能被加载
ES6之前可以使用 函数作用域 和 立即调用函数表达式 将模块定义封装在匿名闭包中
模块永远是单例，无论请求多少次，只会被加载一次

  • CommonJS
    • 在服务器端实现模块化代码组织
    • Node.js 使用了轻微修改版本的 CommonJS，其中它不需要考虑网络延迟问题
    • 使用require(..)指定依赖，用module.exports定义公共API
  • AMD
    • 以浏览器为目标执行环境，需要考虑网络延迟问题
    • 实现核心是用函数包装模块定义，可以防止声明全局变量
    • define(..)定义公共API，异步加载
  • UMD，通用模块定义
    • 统一了CommonJS和AMD生态系统，在启动时检测要使用哪种模块系统，然后进行适当配置，将逻辑包装在一个立即调用的函数表达式里
  • ES6模块
    • <script type=”module”></script>
    • 是异步加载和执行的
    • 顶级this 的值是 undefined（windows下）
    • 解析到上述标签后回立即下载模块文件，但执行会延迟到文档解析完成；有多个标签时也是按顺序执行
    • 嵌入模块的代码不能使用 import 加载到其他模块。只有通过外部文件加载的模块才可以使用 import 加载
    • 使用 export 导出，且必须在模块顶级，不能嵌套在语句中。支持 命名导出 和 默认导出
      • 默认导出使用 default关键字 作为值来导出，default命令只能使用一次，不可重复默认导出。不用加{ }
      • 输出的值是动态绑定的，绑定其所在模块
    • 使用 import 引入其他模块导出的值，也必须出现在模块顶级，不能嵌套
      • 会被提升到模块顶部
      • 导入对模块而言是只读的，相当于是const声明的变量，不能直接修改导出值

总结：

  • CommonJs，由Node.js社区采用
    • require导入模块，module.exports 和 exports 导出模块
    • 在运行时同步加载模块，适用于服务器端（运行时加载）
    • 文件就是一个模块，输入时必须查找对象属性
    • 模块输出的是值的拷贝（浅拷贝），导出值变化不会影响导入值
    • 加载整个模块里的所有方法，然后在使用时再运用对应的方法
  • AMD，RequireJS 库提出
    • require异步加载模块，define定义模块
    • 适用于浏览器端，允许异步加载依赖模块
  • ES6 Modules，ES6中的模块化规范
    • import导入模块，export导出模块
    • 按需加载 异步加载 编译时加载
    • 语法更清晰
    • 输出的是值的引用，import指向内存地址，导入值会随导出值而变化

      • 键名的范围不限于字符串或Symbol，可以是各种类型。可以在键值类型丰富的情况下使用
      • 如果键是简单类型的值，只要值严格相等，包括+0、-0都会视为是一个键
      • 对同一个值多次赋值，后面的将覆盖前面的值
      • 操作方法：get(key) set(key,value) has(key) delete(key) clear()
      • 遍历方法：keys() values() entries() forEach()
      • 可使用扩展运算符…快速转为数组
    • WeakMap与Map相似，但键名只接受对象。
      • 也是弱引用，没有遍历方法和size属性
      • get() set() has() delete() 无法清空，不支持clear()
  • Reflect，与Proxy对象一样，为操作对象而提供的API
    • 将Object上一些明显属于语言层面的方法放到Reflect上
    • 修改某些Object方法的返回结果，使其更合理。如Object.defineProperty(obj,prop,desc)在无法定义属性时会抛出错误，而Reflect.defineProperty…则会返回false
    • Obejct操作从命令变为函数行为，如 in,delete 在 Reflect 中用 has, deleteProperty
    • 方法与Proxy对象 方法一一对应，可以方便地调用对应的Reflect方法完成默认行为
  • Promise，是一个对象，用来传递异步操作的消息
    • 3种状态：Pending Resolved Rejected
    • 从Pending变为Resolved 或 Rejected，一旦发生状态就凝固不会再变
    • 可以将异步操作以同步操作流程表达，避免层层嵌套的回调函数（链式调用，代码更简洁）
    • 缺点：无法中途取消Promise；不设置回调函数那么内部错误不会被反应到外部；Pending状态时无法得知目前在哪个阶段
    • stream模式
      • 连续的数据流：如果你需要处理大量的数据流，例如从文件、网络或其他来源读取连续的数据块，并进行实时处理，那么使用 Stream 模式可能更合适。Stream 提供了一种高效处理大数据的方式，可以分块读取和处理数据，减少内存占用和提高性能。
      • 基于事件驱动：例如实时消息传递、监视文件系统变化等，那么使用 Stream 可能更自然和高效。Stream 提供了一种异步事件驱动的方式，可以方便地处理事件流，并且可以与其他事件驱动的模块和框架无缝集成。
    • Promise 模式相对于 Stream 模式来说，可能更简单和易于理解
      • Promise 提供了一种更直观的方式来处理异步操作，可以通过链式调用来组织和处理异步任务，适合处理一次性的异步操作
      • 根据具体情况，有时 Promise 可能会更容易理解和维护，因为它提供了一种线性的、顺序执行的方式来处理异步任务
  • Generator函数
    • 是一种异步编程解决方案；像一个状态机，封装了多个内部状态
    • 是分段执行的；返回一个遍历器对象
    • 写法：
      • function* funcGenerator(){}
      • 内部使用yield定义不同状态，是暂停执行的标记，本身无返回值
      • 调用next()方法来恢复执行，返回一个对象{value, done}
        • 当调用了该方法后，内部指针指向该语句时才会执行，类似手动地去 惰性求值
        • 可以带参数

函数柯里化Currying，是一种函数式编程的技术

  • 将多参数的函数转换成一系列只接受单个参数的函数的过程。可以理解成为函数预置通用参数，供多次调用
  • 有几个步骤：
    1. 接受原始函数
    2. 转换为一系列函数
    3. 返回最终结果
  • 优点：
    1. 提高函数复用行
    2. 参数延迟求值
    3. 与函数组合使用，可以创建更具灵活性和复用性的函数

序列化

  • 将 数据结构或对象 转换为可以 存储或传输的格式，通常是字节流或文本格式，以便在不同系统之间进行 交换或存储
  • 序列化通常用于将内存中的对象转换为可持久化的形式，比如存储在文件中或通过网络传输
  • 反序列化则是将序列化后的数据重新转换为 原始的数据结构或对象
  • 目的是实现数据持久化、数据传输和跨平台通信
  • JSON、XML等格式都是常见的序列化格式

如何优雅地复用代码

  • 模块化，将功能划分为独立的模块，使用 import和export 来实现模块之前的通信
  • 函数封装，编写可复用的函数，确保函数功能单一性和通用性；（混入mixin、继承）
  • 选择合适的设计模式实现复用
  • 函数式编程，将功能分解为小的纯函数，利用 高阶函数、柯里化 等技术
  • 利用现有库和框架避免重复造轮子；还要编写清晰的文档和注释帮助其他开发者更易理解和复用
  • 测试，利用单元测试来验证代码正确性和稳定性

剩余参数

  • 语法允许我们将一个不定数量的参数表示为一个数组
  • 以减少由参数引起的样板代码

和 arguments对象的区别
主要有三个：

  • 剩余参数只包含 那些没有 对应形参的实参 ，而 arguments 对象包含了 传给函数的所有实参
  • arguments对象只是一个类数组对象，而剩余参数是 真正的Array实例
    • 即能在它上面直接使用所有的数组方法，比如 sort，map，forEach或pop
  • arguments对象还有一些附加的属性（如callee属性）

形参、实参

  • 形参（Parameters）：形式参数是在 函数或方法的定义中声明的变量。
    • 它们在函数或方法内部被用作占位符，用来表示在函数调用时将要传递给函数的值。
    • 形参是函数或方法签名的一部分，它们指定了函数或方法期望接收的数据类型和数量。
  • 实参（Arguments）：实际参数是在函数或方法调用时传递给函数或方法的值。
    • 它们是实际要传递给函数或方法的数据。
    • 当函数或方法被调用时，实参被传递给形参，并且在函数或方法的执行过程中，实参的值将被使用。

数组空值

  • 数组空位有自己的位置，反映在length属性上
  • 但空位的位置上的值未定义，即值不存在。如果一定要读取，结果就是 undefined
  • 那么属性名也不存在，所以无论在 in 、hasOwnProperty、 Object.keys() 方法都取不到空位的key值
  • map遍历时发现空位就直接跳过，不进入回调函数

setTimeout(fn, 0)

  • 常用于让某些代码在当前同步代码块执行完之后再执行，比如等待 DOM 更新完成后执行某些操作。
  • 允许浏览器有时间去处理其他事件（如渲染、处理用户输入等），从而避免长时间阻塞主线程。

前端存储总结

一般的主流方案有三个：

  • Cookie
  • Web Storage
  • IndexedDB

Cookie

1994年被发明，并不是为了在 浏览中存放数据，是为了解决 HTTP协议无状态特性 的问题

HTTP协议无状态特性：用户两次HTTP请求，服务器并不能知道请求来自同一个用户

现在仍在部分领域有价值，但不再适合作为前端本地存储的方案去使用

  • Cookie安全问题

    • 每次请求都会发送Cookie，因为如果不是HTTPS协议并加密，其保存的信息容易被窃取，导致安全风险
    • 一旦信息被窃取，容易被利用，信息泄露
    • 可以用session配合缓解问题，但session同时会占用额外的服务器资源
    • 每次请求自动发送 这个特性会导致CSRF攻击的安全风险
    • 可能还会增加带宽占用，但依据现在的网络环境可以忽略不计

  • Cookie只允许存储4kb大小的数据

  • 操作较复杂，但可以使用类库来解决

常用的有在广告中，跨站标记用户、跟踪用户行为
例如你打开了两个页面，但能看到同样的广告

Cookie 执行同源策略，这里就衍生了第三方Cookie
例如访问 a 时，设置了 b 的 Cookie ，那么对 a 来说就是 第三方Cookie
第三方Cookie有一个重要特性：Cookie可以被服务端通过请求头来要求浏览器设置

画图描述
当用户处于 abc.com 时，浏览器会向 taobao.com/some-ads 发起一个 HTTP 请求
当淘宝服务器返回广告内容时，会顺带一个 Set-Cookie 的 HTTP 请求头，告诉浏览器设置一个源为 taobao.com 的 Cookie，里面存上当前用户的 ID 等信息
这个 Cookie 相对于 abc.com 而言就是第三方 Cookie，因为它属于 taobao.com
而当用户访问 xyz.com 时，由于 xyz.com 上也嵌入了淘宝的广告，因此用户的浏览器也会向 taobao.com/some-ads 发起请求
有意思的来了，发请求时，浏览器发现本地已有 taobao.com 的 Cookie（此前访问 abc.com 时设置的），因此，浏览器会将这个 Cookie 发送过去
淘宝服务器根据发过来的 Cookie，发现当前访问 xyz.com 的用户和之前访问 abc.com 的用户是同一个，因此会返回相同的广告

综上，Cookie 两个特点：可以被浏览器设置、浏览器每次请求会自动带上

  • Cookie设置

    • SameSite：Chrome 80版本新特性，有三个值：Lax（允许部分）/None（都发送）/Strict（不发送）
      默认设置为Lax，即允许同站或子站访问Cookie，目前支持

      get表单
      老版本则是None，允许所有跨站Cookie
      设置Lax可能会导致第三方Cookie失效
      解决：在返回请求的header里设置 Set-Cookie: userId=123; SameSite=None

    • Secure，HttpOnly：常用属性
      Secure只允许 Cookie 在 HTTPS 请求中被使用
      HttpOnly 禁止使用 JS 访问 cookie，这样的好处就是避免了XSS攻击
      配置如下：Set-Cookie: userId=123; SameSite=None; Secure; HttpOnly

  • XSS攻击
    例如一些网站不会对内容的HTML标签进行过滤，那么就可以对其发起攻击或者在console可以用document.cookie获取到cookie信息
    当设置了 HttpOnly 后，以上的做法就失效了

总结：使用场景比较有限，由于安全性和储存空间的问题，一般不推荐使用 Cookie

Web Storage

2014年发布的HTML5标准中，新增了Web Storage，包括LocalStorage、SessionStorage

LocalStorage特点：

  • 使用 Key-Value 形式储存，key、value以字符串存储
    只能以字符串，这个问题可以看这个 https://www.zhihu.com/question/366665675/answer/1127726009
  • 大小有 10MB
  • 使用方便

跟Cookie对比起来：

  • 存储空间更大，使用更方便
  • Cookie可以被服务器设置，但LocalStorage只能被前端手动设置
  • Cookie会由浏览器自动发给服务器，
    LocalStorage需要手动获取放到请求里发送给服务器，因此可以避免 CSRF 攻击

CSRF攻击，跨站伪造攻击

  • 特点：诱导你去访问某个需要你的权限的接口，HTTPS接口并不能阻止这种访问
    某些恶意网站会诱导你点击表单按钮，请求一个你之前登录过的网站的接口，此时浏览器会将你的Cookie一并发送

用LocalStorage替换 比较常见的做法是：
token 加上HTTPS请求，提高登录安全性
即 用户登录后，将服务器返回的 token存到 LocalStorage 里，
之后每次请求前都取出 token放到请求数据里，
服务器就知道是同一个用户发起的请求
由于 HTTPS 的存在，也不用担心 token 会被泄露给第三方

  • 缺点：

    • 无法像Cookie设置过期时间

    • 只能存入字符串，一般使用 JSON.stringify() 配合 JSON.parse() 解决

      但 JSON.stringify() 本身存在一些问题，即无法正确转换JS部分属性，无法转换循环引用的对象
      undefined, Function, RegExp(转换后变成了空对象）, Date(转换后变成了字符串)

      Symbol 无法被转换，
      但由于 Symbol 本身的定义（全局唯一性）就决定了它不应该被转换，否则即使转换回来，也不会是原来的Symbol

      Function 比较特殊，可以先调用 .toString() 转换为字符串储存，需要时再 eval 转回来

      总结：大部分场景下 LocalStorage 可以替代 Cookie ，但最多支持4～5MB，只支持字符串

      不适用 大量数据 和 复杂数据 的存储
