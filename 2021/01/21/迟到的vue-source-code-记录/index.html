迟到的vue source code 记录

前期准备：需要拉取远程vue代码

package.json

构建后的主文件: dist/vue.runtime.common.js
构建入口js: scripts/config.js
我们主要看platform: web 的目标构建对象 TARGET: web-full-dev

build.js

过滤构建配置，构建不同用途的vue.js

config.js

找到web-full-dev
可得知入口为 web/entry-runtime-with-compiler.js
输出文件 dist/vue.js

web/entry-runtime-with-compiler.js

得知Vue 来自 runtime/index => core/index => instance/index

core/instance/index.js

一层层剥开后找到的最终奥义，看完有种 啊，终于 的感觉

也就是说 Vue实际上是一个类，在js中用Function实现，这是为啥需要 new Vue()来初始化了
接下来就调用 ① init，该方法在initMixin方法中定义

core/instance/init.js

抓几个关键点来看，init方法主要做了

  • 合并配置
  • 初始化生命周期
  • 初始化事件
  • 渲染
  • data、props、computed、watch等

最后，如果检测到有el属性，就调 ② $mount挂载vm

感觉线索断了？再回到 entry-runtime-with-compiler看一眼

web/entry-runtime-with-compiler.js

原来$mount在这里定义
先缓存了vue原型上的$mount，再重新定义该方法
传入两个可选参数：

  • el（挂载元素 string | Element）
  • hydrating (ssr时为true)

返回类型为Component，调用前面缓存的mount

el不可挂载到 html或body 元素上
如果没有render方法，则将template/el转为render function

那render方法是怎么来的呢 =>
const { render, staticRenderFns } = compileToFunctions()

compileToFunctions也被赋值给了Vue.compile

也就是说，③ 先编译compile 后 才得到 ④ render方法

来看看compileToFunctions，源码弯弯绕绕的，那这里就不贴了
源码做的工作就是将 template转为 字符串 返回给 render属性

上面提到先缓存了vue原型上的$mount，那么这个是在哪里定义的呢？

web/runtime/index.js

之所以这么设计完全是为了复用，因为它是可以被runtime only版本的Vue直接使用的。

最后实际调用了mountComponent，那我们来看一下mountComponent方法

core/instance/lifecycle.js

可以看到 核心逻辑就是 实例化一个watcher，回调 updateComponent方法
在该方法中调用 vm._render() 生成并返回VNode，最终调用 vm._update更新DOM

再回到 core/instance/index.js ，看看_render方法

core/instance/render.js

看到最后，该方法返回 ⑤ VNode
来看下VNode，它是对真实DOM的一种抽象描述，用来映射对真实DOM的渲染，并不包含DOM操作，非常地轻量简便

core/vdom/vnode.js

也就是说，Vue用 Vnode类 去描述一个 Virtual DOM，包含了tag, data, children等属性
Vue.js 中 Virtual DOM 是借鉴了 开源库snabbdom 的实现

再回到render方法，发现传参中有一个createElement
这个是什么呢？来看一下

core/vdom/create-element.js

这里主要看两个方法
normalizationType 表示子节点规范的类型

  • simpleNormalizeChildren
    理论上编译生成的children都是Vnode
    当一个函数式组件返回数组而不是根节点时，通过这个方法来扁平化children数组
    保证只有一层深度
    因为函数式组件已经规范化他们的子节点了

这能作为性能优化的一个点

  • normalizeChildren
    当子节点有 嵌套数组 时，例如 <template>, <slot>, v-for, 用户手写的render函数
    需要下面这个方法来规范化children

normalizeArrayChildren 主要逻辑就是 遍历children，规范化每个节点

  • 单个节点的类型如果是数组，递归调用 normalizeArrayChildren
  • 如果是基础类型，调用 createTextVNode 转为 VNode
  • 如果存在两个 连续的textNode，则合并这两个节点
  • 如果有嵌套数组节点，则用 nestedIndex 更新节点key
  • 最后返回一个类型为 VNode 的数组

那么，规范化完children后就去 实例化VNode

接下来再看看之前提到的vm._update方法

core/instance/lifecycle.js

可以看到 初始化渲染的关键就在 patch函数
那我们来找下patch的具体实现

platforms/web/runtime/patch.js

createPatchFunction方法包含 nodeOps 和 modules
其中，nodeOps 封装了一系列 DOM 操作的方法
modules 定义了一些模块的 钩子函数 的实现
看下 createPatchFunction 的具体实现⬇️

core/vdom/patch.js

函数最终返回一个 patch 方法，即 vm.patch 调用的就是该方法
看到这里，可以看到patch主要逻辑相同，差异在入参
通过 createPatchFunction 先传入不同平台的参数，再赋给patch
这样 patch 对外是 无差别的，这种编程技巧非常值得学习
